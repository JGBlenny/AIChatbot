# 知識完善迴圈系統 - 完整操作流程

## 系統概述

知識完善迴圈系統是一個自動化的知識庫優化流程，透過「回測 → 分析 → 生成 → 審核 → 驗證」的閉環，持續提升 AI 回答品質。

---

   ├─ 設定批次大小：50 題
   ├─ 設定目標通過率：85%
   └─ 系統自動選取固定測試集（分層隨機抽樣）

   ↓

第 2 步：執行迭代（自動執行 8 步驟）
   ↓
   [迴圈管理 Tab] → 點擊迴圈詳情 → 點擊「執行迭代」

   ├─ Step 1: 執行回測（50 題）
   │   └─ 調用 RAG API 測試所有題目

   ├─ Step 3: 智能分類（OpenAI）
   │   └─ 分類為：SOP知識/表單填寫/系統配置/API查詢
   ├─ Step 4-5: 按類別聚類
   │   └─ 將相似問題聚類在一起
   ├─ Step 6: 生成知識（OpenAI）
   │   └─ 為每個聚類生成知識/SOP
   ├─ Step 7: 寫入待審核表
   │   └─ 知識寫入 loop_generated_knowledge
   └─ Step 8: 更新迴圈狀態 → REVIEWING

   ↓

第 3 步：人工審核知識
   ↓
   [知識審核 Tab]
   ├─ 查看待審核知識清單
   ├─ 檢查重複警告（相似度檢測）
   ├─ 逐一審核或批量審核
   │   ├─ 批准 → 立即同步到正式知識庫
   │   └─ 拒絕 → 標記為 rejected
   └─ 審核完成後，準備執行下一次迭代

   ↓

第 4 步：執行下一次迭代
   ↓
   [迴圈管理 Tab] → 點擊「執行迭代」（第 2 次）
   └─ 重複 Step 1-8，測試同樣的 50 題
       檢查通過率是否提升

   ↓

第 5 步：檢查是否達成目標
   ↓
   [回測結果 Tab] → 查看迭代對比
   ├─ 第 1 次迭代：通過率 60% (30/50)
   ├─ 第 2 次迭代：通過率 74% (37/50) ↑ 14%
   ├─ 第 3 次迭代：通過率 78% (39/50) ↑ 4%
   └─ 目標通過率：85%（尚差 7%）

   判斷：
   ├─ 已達標（≥85%）→ 完成批次
   └─ 未達標 → 繼續迭代（最多 10 次）

   ↓

第 6 步：完成批次
   ↓
   [迴圈管理 Tab] → 點擊「完成批次」
   └─ 標記迴圈為 COMPLETED

   ↓

第 7 步：啟動下一批次（可選）
   ↓
   [迴圈管理 Tab] → 點擊「啟動下一批次」
   └─ 自動選取新的 50 題（避免與第 1 批重複）
       重複流程
```

## 詳細操作步驟

### 步驟 1：啟動新迴圈

**位置**：`迴圈管理` Tab

**操作**：
1. 點擊「啟動新迴圈」按鈕
2. 填寫表單：
   ```
   迴圈名稱：第1批-包租業
   業者：富喬物業管理顧問股份有限公司 (vendor_id=2)
   批次大小：50 題
   最大迭代次數：10 次
   目標通過率：85%
   ```
3. 點擊「確認啟動」

**系統執行**：
- 建立新的迴圈記錄（`knowledge_completion_loops` 表）
- 從 `test_scenarios` 表選取 50 題（分層隨機抽樣）
- 記錄固定測試集 ID 到 `scenario_ids` 欄位
- 狀態設為 `PENDING`

**結果**：
- 迴圈列表新增一筆記錄
- 顯示迴圈 ID（例如：#3）

---

### 步驟 2：執行迭代

**位置**：`迴圈管理` Tab → 點擊迴圈詳情

**操作**：
1. 在迴圈列表中找到剛建立的迴圈
2. 點擊「詳情」按鈕，打開迴圈詳情 Modal
3. 點擊「執行迭代」按鈕
4. 確認執行（選擇非同步模式）

**系統執行（自動）**：

```
第 1 階段：回測執行（5-10 分鐘）
├─ 狀態：BACKTESTING
├─ 調用 RAG API 測試 50 題
├─ 記錄結果到 backtest_results 表
└─ 計算通過率（例如：60%）

第 2 階段：失敗分析（< 1 分鐘）
├─ 狀態：ANALYZING
├─ 篩選失敗的 20 題
└─ 收集問題、期望答案、實際答案

第 3 階段：智能分類（1-2 分鐘）
├─ 狀態：ANALYZING
├─ 調用 OpenAI API 分類 20 個失敗案例
└─ 分類結果：
    ├─ sop_knowledge: 8 題（SOP 流程知識）
    ├─ form_fill: 5 題（表單填寫）
    ├─ system_config: 4 題（系統配置）
    └─ api_query: 3 題（不生成靜態知識）

第 4-5 階段：聚類（< 1 分鐘）
├─ 狀態：ANALYZING
├─ 按類別分離：
│   ├─ SOP 問題：13 題（sop_knowledge + form_fill）
│   └─ 知識問題：4 題（system_config）
└─ 分別聚類：
    ├─ SOP 聚類：3 個聚類
    └─ 知識聚類：2 個聚類

第 6 階段：生成知識（2-5 分鐘）
├─ 狀態：GENERATING
├─ 對每個聚類調用 OpenAI API 生成知識
└─ 生成結果：
    ├─ 3 個 SOP 項目 → 寫入 loop_generated_knowledge
    └─ 2 個知識項目 → 寫入 loop_generated_knowledge

第 7 階段：寫入待審核表（< 1 秒）
├─ 狀態：GENERATING
├─ 寫入 5 筆記錄到 loop_generated_knowledge
├─ 狀態：pending
└─ 執行重複檢測（pgvector 向量搜尋）

第 8 階段：完成迭代（< 1 秒）
├─ 狀態：REVIEWING
├─ 更新 current_iteration = 1
├─ 記錄執行日誌到 loop_execution_logs
└─ 等待人工審核
```

**前端顯示**：
- 進度條顯示執行階段
- 預估剩餘時間
- 完成後狀態變為「審核中」

## 關鍵概念說明

### 1. 固定測試集

**目的**：確保同一迴圈的所有迭代使用相同測試題目，才能比較改善幅度。

**實作**：
- 迴圈啟動時選取 50 題
- 記錄到 `scenario_ids` 欄位：`[1, 5, 12, 18, ...]`
- 每次迭代都測試這 50 題
- 通過率變化才有意義

**範例**：
```
迭代 1：測試題目 [1,5,12,18,...] → 通過率 60%
迭代 2：測試題目 [1,5,12,18,...] → 通過率 74% ↑ 14%
迭代 3：測試題目 [1,5,12,18,...] → 通過率 78% ↑ 4%
```

如果每次迭代測試不同題目，通過率無法比較：
```
迭代 1：測試題目 [1,5,12] → 通過率 60%
迭代 2：測試題目 [20,30,40] → 通過率 80% ❌（無法比較）
```

---

### 2. 分層隨機抽樣

**目的**：確保測試集涵蓋不同難度，避免測試偏差。

**實作**：
```
總題庫：500 題
├─ 簡單：100 題（20%）
├─ 中等：250 題（50%）
└─ 困難：150 題（30%）

抽樣 50 題：
├─ 簡單：10 題（20% * 50）
├─ 中等：25 題（50% * 50）
└─ 困難：15 題（30% * 50）
```

**好處**：
- 測試集有代表性
- 避免只測試簡單題（通過率虛高）
- 避免只測試困難題（通過率虛低）

---

### 3. 批次與迴圈

**批次**：
- 一組測試題目（例如 50 題）
- 對應一個迴圈 ID

**迴圈**：
- 在同一批次上反覆迭代
- 目標：提升通過率到 85%

**範例**：
```
第 1 批次（迴圈 #3）
├─ 題目：1-50
├─ 迭代 1：60% → 生成 5 個知識
├─ 迭代 2：74% → 生成 3 個知識
├─ 迭代 3：78% → 生成 2 個知識
└─ 迭代 4：88% ✅ 達標

第 2 批次（迴圈 #4）
├─ 題目：51-100（避開 1-50）
├─ 迭代 1：65%
└─ ...
```

---

### 4. 知識審核與立即同步

**流程**：
```
生成知識
  ↓
寫入 loop_generated_knowledge（status=pending）
  ↓
人工審核
  ↓
批准 → 立即同步到正式庫
  ├─ SOP → vendor_sop_items（status=approved）
  └─ 知識 → knowledge_base（status=approved）
  ↓
RAG 查詢立即生效（無需重啟）
```

**重點**：
- 審核通過的知識**立即生效**
- 下一次迭代會使用新知識回答
- 通過率應該會提升

---

### 5. 重複檢測

**目的**：避免生成重複的知識。

**實作**：
```
生成新知識時
  ↓
生成問題 embedding（向量）
  ↓
pgvector 向量搜尋（找相似知識）
  ↓
計算相似度
  ├─ ≥ 0.95：幾乎重複，建議拒絕
  ├─ ≥ 0.85：高度相似，人工判斷
  └─ < 0.85：不相似，安全
  ↓
前端顯示警告
```

**範例**：
```
⚠️ 重複警告
檢測到 1 個高度相似的知識（相似度 93%）
- [knowledge_base #456] 租金繳納日期說明

建議：請人工判斷是否為重複內容
```

## 常見問題 (FAQ)

### Q1: 為什麼需要多次迭代？
**A**: 單次生成的知識無法涵蓋所有失敗案例。透過多次迭代：
- 第 1 次：解決明顯的知識缺口（通過率 60% → 74%）
- 第 2 次：優化不完整的回答（通過率 74% → 78%）
- 第 3 次：處理邊緣案例（通過率 78% → 88%）

---

### Q2: 為什麼要分批處理？
**A**:
- 每批 50 題，聚焦於特定主題
- 避免一次處理 500 題（太耗時、難以審核）
- 完成第 1 批後，啟動第 2 批（避免重複題目）

---

### Q3: 如何判斷何時停止迭代？
**A**:
- **達成目標**：通過率 ≥ 85%
- **超過次數**：迭代次數 ≥ 10 次
- **邊際遞減**：連續 2 次迭代改善 < 2%

---

### Q4: 驗證回測 Tab 有什麼用？
**A**: （此功能目前為 Beta，尚未實作）
- 快速驗證知識改善效果
- 只測試失敗案例 + 抽樣通過案例（節省時間）
- 檢測 Regression（新知識是否導致原本通過的題目失敗）

目前可透過「執行完整迭代」來驗證效果。

---

### Q5: 如何查看某個迴圈的所有回測結果？
**A**:
1. 複製迴圈 ID（例如：#3）
2. 切換到「回測結果」Tab
3. URL 加上參數：`/backtest?tab=backtest-results&loopId=3`
4. 系統自動載入該迴圈的迭代對比

## 實際範例

**Day 2：繼續優化**

      ├─ 生成 3 個知識項目
      └─ 狀態：REVIEWING

14:00 審核 3 個知識
      └─ 批准 3 個

14:30 執行第 4 次迭代
      ├─ 通過率：78% → 88% ↑ 10% ✅
      └─ 達成目標！

15:00 完成批次
```

**Day 3：啟動第 2 批次**
```
09:00 啟動下一批次
      ├─ 名稱：第2批-包租業
      ├─ 系統自動選取新的 50 題（避開第1批）
      └─ 重複流程
```

## 總結

### 核心流程
```
啟動迴圈 → 執行迭代 → 審核知識 → 查看結果 → 判斷是否達標
    ↓           ↓          ↓          ↓            ↓
 選取50題    自動8步驟   批准/拒絕   迭代對比    達標→完成
                                                  未達標→繼續迭代
```

### 關鍵原則
1. **固定測試集**：同一迴圈的所有迭代使用相同題目
2. **分層抽樣**：確保測試集涵蓋不同難度
3. **立即同步**：審核通過的知識立即生效
4. **批次分離**：避免不同批次選取重複題目
5. **重複檢測**：避免生成重複知識
