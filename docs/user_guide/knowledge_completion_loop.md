## 🎯 什麼是知識完善迴圈？

知識完善迴圈是一個**迭代式的知識優化系統**，透過以下 8 個步驟持續改善知識庫品質：

```
1. 執行回測 → 2. 分析失敗案例 → 3. 智能分類 → 4. 按類別分離 →
5. 分別聚類 → 6. 生成知識 → 7. 人工審核 → 8. 檢查通過率並決定是否迭代
```

### 核心概念

- **迴圈（Loop）**：一組固定的測試情境（例如 50 題），對應一個 `loop_id`
- **迭代（Iteration）**：在同一迴圈內反覆執行回測、生成知識、優化流程
- **批次（Batch）**：等同於迴圈，用於區分不同的測試集
- **固定測試集**：迴圈啟動時選取的測試情境，所有迭代使用相同測試集
- **目標通過率**：預期達成的測試通過率（預設 85%）

### 為什麼需要固定測試集？

- ✅ **確保可比較性**：每次迭代測試相同的問題，才能準確評估改善幅度
- ✅ **避免隨機誤差**：不同測試集可能有不同難度，固定測試集消除這種干擾
- ✅ **追蹤進度**：清楚看到知識優化的效果（例如：第 1 次迭代通過率 60% → 第 3 次迭代 85%）

## 🚀 快速開始

### 使用場景 1: 快速驗證（50 題）

**適用情境**：
- 測試新的知識生成策略
- 驗證知識改善效果
- 快速迭代優化

**步驟**：
1. 點擊「啟動新迴圈」
2. 填寫表單：
   - 迴圈名稱：「快速驗證 2026-03-27」
   - 批次大小：**50**
   - 目標通過率：**85%**
   - 最大迭代次數：**5**
3. 點擊「啟動迴圈」
4. 等待固定測試集選取完成（約 5 秒）
5. 點擊「執行迭代」
6. 等待迭代完成（約 10-15 分鐘）
7. 進入「知識審核」Tab，批量審核生成的知識
8. 重複步驟 5-7，直到通過率達標

**預期時間**：
- 單次迭代：10-15 分鐘
- 完整流程（5 次迭代）：50-75 分鐘

## 📋 完整流程說明

### Step 1: 啟動迴圈

#### 1.1 進入迴圈管理頁面

- 路由：`/knowledge-completion/loops`
- Tab：「迴圈管理」

#### 1.2 點擊「啟動新迴圈」按鈕

![啟動新迴圈按鈕](screenshots/start_loop_button.png)

#### 1.3 填寫迴圈配置表單

| 欄位 | 說明 | 必填 | 預設值 | 範圍 |
|------|------|------|--------|------|
| 迴圈名稱 | 迴圈的名稱，建議包含日期 | ✅ | - | 1-200 字元 |
| 批次大小 | 測試情境數量 | ✅ | 50 | 1-3000 |
| 目標通過率 | 預期達成的通過率 | ✅ | 0.85 | 0.0-1.0 |
| 最大迭代次數 | 最多執行幾次迭代 | ✅ | 10 | 1-50 |
| 選取策略 | 測試情境選取策略 | ❌ | 分層隨機抽樣 | 3 種 |
| 難度分布 | 各難度的比例 | ❌ | 簡單 20%, 中等 50%, 困難 30% | - |
| 預算限制 | OpenAI API 成本上限（USD） | ❌ | 無限制 | ≥ 0 |
| 父迴圈 ID | 關聯的父迴圈（批次鏈） | ❌ | 無 | - |

**選取策略說明**：
- **分層隨機抽樣**（推薦）：按難度分層，每層按比例隨機選取，確保涵蓋不同難度
- **順序選取**：按 ID 順序選取前 N 題
- **完全隨機**：完全隨機選取 N 題

**難度分布說明**：
- 預設比例：簡單 20%, 中等 50%, 困難 30%
- 可自訂比例，總和必須為 100%
- 僅在「分層隨機抽樣」策略時有效

#### 1.4 點擊「啟動迴圈」

系統將：
1. 建立新的迴圈記錄
2. 選取固定測試集（分層隨機抽樣）
3. 初始化成本追蹤器
4. 返回迴圈詳情頁面

**預期回應時間**：< 1 秒

### Step 2: 執行迭代

#### 2.1 進入迴圈詳情頁面

- 路由：`/knowledge-completion/loops/:id`
- 顯示迴圈狀態、固定測試集、迭代歷史

#### 2.2 點擊「執行迭代」按鈕

![執行迭代按鈕](screenshots/execute_iteration_button.png)

#### 2.3 選擇執行模式

| 模式 | 說明 | 適用場景 | 回應方式 |
|------|------|---------|---------|
| **非同步模式**（推薦） | 背景執行，前端輪詢狀態 | 長時間迭代（≥ 10 分鐘） | 立即返回 task_id |
| **同步模式** | 等待執行完成後返回 | 快速測試（< 5 分鐘） | 等待完成後返回結果 |

**預設模式**：非同步模式

#### 2.4 監控執行進度（非同步模式）

前端每 5 秒輪詢迴圈狀態（`GET /api/v1/loops/:id`）：

**階段 1: 執行回測（10-120 分鐘）**
- 狀態：`backtesting`
- 進度顯示：「正在測試第 23/50 個情境...」
- 進度百分比：0-100%

**階段 2: 分析失敗案例（< 1 分鐘）**
- 狀態：`analyzing`
- 進度顯示：「正在分析失敗案例...」

**階段 3: 智能分類（1-5 分鐘）**
- 狀態：`classifying`
- 進度顯示：「正在分類知識缺口（OpenAI）...」

**階段 4: 生成知識（2-10 分鐘）**
- 狀態：`generating`
- 進度顯示：「正在生成知識（第 5/10 個聚類）...」

**階段 5: 完成**
- 狀態：`reviewing`
- 進度顯示：「迭代完成，請前往審核中心審核知識」

#### 2.5 迭代完成通知

- 前端檢測到狀態為 `reviewing`，停止輪詢
- 顯示通知：「✅ 迭代 1 完成！生成了 15 個知識項目，請前往審核中心審核」
- 提供「前往審核」按鈕，跳轉到審核中心

### Step 3: 審核知識

#### 3.1 進入審核中心

- 路由：`/knowledge-completion/review-center`
- Tab：「知識審核」

#### 3.2 查看待審核知識列表

**篩選條件**：
- 迴圈：選擇特定迴圈（或全部）
- 知識類型：SOP / 一般知識
- 重複警告：有警告 / 無警告
- 狀態：待審核 / 已批准 / 已拒絕

**顯示欄位**：
- ID
- 迴圈名稱
- 迭代次數
- 問題（摘要）
- 知識類型（SOP / 一般知識）
- 重複警告（⚠️  高度相似 93%）
- 狀態（待審核 / 已批准 / 已拒絕）
- 操作按鈕

#### 3.3 重複檢測警告說明

系統使用 **pgvector 向量相似度搜尋**檢測重複知識：

| 相似度等級 | 閾值 | 圖標 | 建議操作 |
|-----------|------|------|---------|
| **幾乎完全相同** | ≥ 0.95 | 🔴 | **拒絕**（避免重複） |
| **高度相似** | ≥ 0.85 | 🟡 | **人工判斷**（檢視相似知識詳情） |
| **相關知識** | ≥ 0.75 | 🔵 | **參考**（可能需要合併或調整） |
| **無相似** | < 0.75 | - | **批准**（新知識） |

**查看相似知識詳情**：
- 點擊知識項目，展開詳情面板
- 顯示「相似知識」區塊：
  - 知識來源（knowledge_base / vendor_sop_items / loop_generated_knowledge）
  - 知識 ID
  - 問題摘要
  - 相似度評分（例如：93%）
  - 「查看完整內容」按鈕

#### 3.4 單一審核操作

**操作步驟**：
1. 點擊知識項目右側的「審核」按鈕
2. 展開審核面板
3. 檢視知識內容：
   - 問題（完整）
   - 答案（完整）
   - 知識類型（SOP / 一般知識）
   - SOP 配置（如果是 SOP）
   - 相似知識警告（如果有）
4. 選擇操作：
   - **批准**：知識正確，同步到正式庫
   - **拒絕**：知識錯誤或重複
   - **修改後批准**：編輯知識內容後批准
5. 填寫審核備註（可選）
6. 點擊「確認」

**審核後自動操作**：
- **批准**：
  1. 立即同步到正式知識庫（`knowledge_base` 或 `vendor_sop_items`）
  2. 生成 embedding 向量（調用 Embedding API）
  3. 更新知識狀態為 `synced`
  4. 記錄審核事件到 `loop_execution_logs`
  5. RAG 查詢立即生效（無需重啟服務）
- **拒絕**：
  1. 更新知識狀態為 `rejected`
  2. 記錄拒絕原因
  3. 知識不會同步到正式庫

### Step 4: 批量審核知識

#### 4.1 篩選知識

使用篩選條件縮小範圍：
- 迴圈：選擇特定迴圈
- 知識類型：一般知識（批量審核 SOP 需謹慎）
- 重複警告：**無警告**（確保不批量批准重複知識）

#### 4.2 全選篩選結果

- 點擊表格左上角的「全選核取方塊」
- 或點擊「全選篩選結果」按鈕

**全選模式**：
- **當前頁全選**：僅選取當前頁的知識項目（例如 20 個）
- **所有頁全選**：選取所有符合篩選條件的知識項目（例如 50 個）

#### 4.3 查看批量操作工具列

工具列顯示：
- 已選取數量：「已選取 20 個項目」
- 知識類型分布：「SOP: 5 個, 一般知識: 15 個」
- 重複警告統計：「有警告: 2 個, 無警告: 18 個」
- 批量操作按鈕：「批量批准」、「批量拒絕」

#### 4.4 點擊「批量批准」

#### 4.5 確認對話框

顯示統計資訊：
- 總數：20 個
- 知識類型分布：SOP: 5 個, 一般知識: 15 個
- 重複警告統計：有警告: 2 個, 無警告: 18 個
- ⚠️  警告提示：「檢測到 2 個知識有重複警告，建議先移除這些項目再批量批准」

**確認選項**：
- 「移除有警告的項目」（推薦）
- 「全部批准」（不推薦）
- 「取消」

#### 4.6 批量處理進度

顯示進度對話框：
- 進度條：0-100%
- 當前處理：「正在處理第 5/20 個知識...」
- 已處理列表：
  - ✅ [知識 #123] 租金繳納日期（成功）
  - ✅ [知識 #124] 停車位申請流程（成功）
  - ❌ [知識 #125] 垃圾收取規範（失敗：向量生成錯誤）
  - ...

#### 4.7 結果摘要對話框

顯示批量處理結果：
- 總數：20 個
- 成功：18 個
- 失敗：2 個
- 失敗項目列表：
  - [知識 #125] 垃圾收取規範 - 錯誤：向量生成錯誤
  - [知識 #130] 社區公設使用規範 - 錯誤：資料庫連接超時
- 耗時：12.5 秒
- 操作按鈕：「重試失敗項目」、「關閉」

**部分成功模式說明**：
- ✅ 成功的項目已同步到正式庫，RAG 查詢立即生效
- ❌ 失敗的項目保持 `pending` 狀態，可稍後重試
- ⚠️  一個失敗不影響其他項目

### Step 5: 驗證效果回測（可選功能）

**⚠️  注意**：此功能為可選功能，優先實作核心流程。如需使用，請聯繫開發團隊。

#### 5.1 何時使用驗證回測？

- ✅ **快速驗證知識改善效果**：不需要執行完整迭代（測試所有 50 題）
- ✅ **Regression 檢測**：檢測新增知識是否導致其他問題惡化
- ✅ **高品質要求**：審核通過的知識需要再次驗證後才同步到正式庫

#### 5.2 驗證範圍選項

| 驗證範圍 | 說明 | 適用場景 |
|---------|------|---------|
| **僅測試失敗案例** | 只測試上次迭代失敗的測試案例 | 快速驗證知識改善效果 |
| **測試所有案例** | 測試所有固定測試集 | Regression 檢測 |
| **失敗案例 + 抽樣通過案例** | 失敗案例 + 20% 隨機抽樣通過案例 | 平衡效率與 Regression 檢測 |

#### 5.3 執行驗證回測

1. 點擊「驗證效果回測」按鈕
2. 選擇驗證範圍
3. 點擊「執行驗證」
4. 等待驗證完成（約 5-20 分鐘，視範圍而定）
5. 查看驗證結果

#### 5.4 驗證結果判斷

**驗證通過條件**：
- 改善幅度 ≥ 5% **或** 通過率 ≥ 70%
- **且** 無 Regression（通過案例沒有變成失敗）

**驗證結果顯示**：
- 通過率：75% → 82% (+7%)
- 改善幅度：+7%
- Regression 檢測：無（0 個通過案例變成失敗）
- 驗證結果：✅ 通過

**驗證失敗處理**：
- 知識狀態標記為 `need_improvement`
- 建議：調整知識內容或重新生成

### Step 6: 檢查通過率並決定是否迭代

#### 6.1 查看迭代歷史

- 路由：`/knowledge-completion/loops/:id`
- Tab：「迭代歷史」

**顯示內容**：
- 迭代次數
- 執行時間
- 通過率（例如：60% → 75% → 85%）
- 新增知識數量
- 同步知識數量
- 耗時

#### 6.2 檢查通過率

**通過率趨勢圖**（ECharts）：
- X 軸：迭代次數（1, 2, 3, ...）
- Y 軸：通過率（0-100%）
- 目標線：85%（紅色虛線）
- 數據線：實際通過率（藍色實線）

#### 6.3 決定下一步

**情況 1: 通過率已達標（≥ 85%）**
- ✅ 點擊「完成批次」按鈕
- 系統標記迴圈為 `COMPLETED`
- 顯示完成摘要：
  - 總迭代次數：3 次
  - 最終通過率：87%
  - 總新增知識：25 個
  - 總同步知識：20 個
  - 總耗時：45 分鐘
  - 總成本：$5.50 USD

**情況 2: 通過率未達標（< 85%）且未超過最大迭代次數**
- ⚠️  點擊「執行迭代」按鈕，開始下一次迭代
- 重複 Step 2-6

**情況 3: 通過率未達標但已達最大迭代次數**
- 🚨 系統自動標記迴圈為 `FAILED`
- 建議：
  - 調整測試情境難度分布
  - 增加批次大小
  - 增加最大迭代次數
  - 檢視失敗案例的失敗原因

### Step 7: 啟動下一批次（批次鏈）

**適用場景**：
- 完成第一批次（50 題），想繼續測試另外 50 題
- 需要避免批次間重複選取測試情境

#### 7.1 點擊「啟動下一批次」按鈕

#### 7.2 填寫表單（自動關聯父迴圈）

| 欄位 | 說明 | 自動填入 |
|------|------|---------|
| 迴圈名稱 | 第二批次名稱 | 「第二批次 - {父迴圈名稱}」 |
| 批次大小 | 測試情境數量 | 50（與父迴圈相同） |
| 目標通過率 | 預期通過率 | 0.85（與父迴圈相同） |
| 最大迭代次數 | 最多迭代次數 | 10（與父迴圈相同） |
| **父迴圈 ID** | 關聯的父迴圈 | **自動填入（不可編輯）** |

#### 7.3 系統自動排除已使用的測試情境

- 查詢父迴圈的 `scenario_ids`：[1, 2, 3, ..., 50]
- 選取測試情境時排除這些 ID
- 確保第二批次的測試情境為：[51, 52, 53, ..., 100]

#### 7.4 開始第二批次迭代

- 重複 Step 2-6

### 1. 批次大小選擇

|---------|---------|------------|-------------|
| **10-50** | 快速驗證、緊急修復 | 5-15 分鐘 | 5-15 個 |
| **50-200** | 日常優化、週期性改善 | 15-30 分鐘 | 15-50 個 |
| **200-500** | 季度評估、全面優化 | 30-90 分鐘 | 50-150 個 |
| **500-3000** | 年度大檢查、新業者建置 | 90-120 分鐘 | 150-500 個 |

### 2. 目標通過率設定

| 目標通過率 | 適用場景 | 難度 |
|-----------|---------|------|
| **70-80%** | 初期建置、困難問題集 | 中等 |
| **80-90%** | 標準要求、一般問題集 | 一般 |
| **90-95%** | 高品質要求、簡單問題集 | 較高 |
| **95-100%** | 極端高品質要求 | 極高（可能無法達成） |

**建議**：
- 首次使用建議設定 **85%**（平衡品質與效率）
- 視實際情況調整（如果 3 次迭代仍未達標，可降低目標）

### 3. 預算控制建議

| 批次大小 | 建議預算（USD） | 說明 |
|---------|----------------|------|
| 50 | $10-20 | 單次迭代約 $2-4 |
| 500 | $50-100 | 單次迭代約 $10-20 |
| 3000 | $200-500 | 單次迭代約 $50-100 |

**預算計算公式**：
```
預算 = 批次大小 × 每題成本 × 迭代次數
每題成本 ≈ $0.05-0.10（含回測、分類、生成）
```

### 4. 批量審核最佳實踐

**✅ 建議做法**：
- 先篩選「無重複警告」的知識，批量批准
- 有重複警告的知識，單一審核人工判斷
- 批量審核前先審核 2-3 個樣本，確認品質
- 批量審核數量建議 ≤ 50 個/次（避免超時）

**❌ 不建議做法**：
- 直接全選所有知識批量批准（包含有重複警告的）
- 批量審核 SOP 知識（SOP 需謹慎審核）
- 批量審核數量 > 100 個（可能超時或部分失敗）

## ❓ 常見問題解答（FAQ）

### Q1: 為什麼我的迭代執行了 30 分鐘還沒完成？

**可能原因**：
1. 批次大小太大（例如 3000 題需要 90-120 分鐘）
2. OpenAI API 速率限制（重試機制導致延遲）
3. 回測 API 效能問題（RAG 查詢延遲）

**解決方法**：
- 查看迴圈詳情頁面的「進度顯示」，確認當前階段
- 如果長時間卡在某個階段，聯繫技術支援
- 下次使用較小的批次大小（例如 50 題）

---

### Q2: 批量審核失敗了怎麼辦？

**部分成功模式**：
- ✅ 成功的項目已同步到正式庫
- ❌ 失敗的項目保持 `pending` 狀態

**處理步驟**：
1. 查看「結果摘要對話框」中的失敗項目列表
2. 記錄失敗原因（例如：向量生成錯誤、資料庫連接超時）
3. 點擊「重試失敗項目」按鈕
4. 如果重試仍失敗，聯繫技術支援

---

### Q3: 如何判斷知識是否重複？

**自動判斷（重複檢測系統）**：
- 🔴 **相似度 ≥ 95%**：幾乎完全相同，建議拒絕
- 🟡 **相似度 ≥ 85%**：高度相似，人工判斷
- 🔵 **相似度 ≥ 75%**：相關知識，參考即可

**人工判斷步驟**：
1. 點擊知識項目，查看「相似知識」區塊
2. 點擊「查看完整內容」，比對問題與答案
3. 判斷是否為：
   - **完全重複**：問題、答案完全相同 → 拒絕
   - **部分重複**：問題相同，答案補充或調整 → 考慮合併或調整
   - **相關但不重複**：問題相似但不同，答案不同 → 批准

---

### Q4: 迴圈狀態一直是「審核中」，為什麼無法執行下一次迭代？

**原因**：
- 迴圈狀態為 `reviewing`，需要先審核知識

**解決方法**：
1. 進入「知識審核」Tab
2. 批量審核待審核知識（或全部拒絕）
3. 審核完成後，狀態會自動變回 `running`
4. 此時可執行下一次迭代

---

### Q5: 如何查看 OpenAI API 成本？

**查看單次迴圈成本**：
1. 進入迴圈詳情頁面
2. 查看「基本資訊」Tab
3. 顯示欄位：
   - 總成本：$5.50 USD
   - 預算限制：$100.00 USD
   - 預算使用率：5.5%

**查看歷史成本**：
1. 進入「迴圈列表」頁面
2. 查看「總成本」欄位
3. 可按成本排序

**成本分解**：
- 回測：$0.00（調用內部 RAG API，無額外成本）
- 智能分類：$0.50-1.00（OpenAI GPT-4o-mini）
- 知識生成：$4.00-5.00（OpenAI GPT-4o-mini）
- 向量生成：$0.50-1.00（Embedding API）

---

### Q6: 批次間如何避免重複選取測試情境？

**自動機制**：
- 使用「父迴圈 ID」關聯批次
- 系統自動排除父迴圈的 `scenario_ids`

**操作步驟**：
1. 完成第一批次（例如 loop_id = 1，scenario_ids = [1, 2, ..., 50]）
2. 點擊「啟動下一批次」按鈕
3. 系統自動填入 `parent_loop_id = 1`
4. 選取測試情境時，排除 [1, 2, ..., 50]
5. 第二批次的測試情境為 [51, 52, ..., 100]

---

### Q7: 迴圈執行失敗了怎麼辦？

**常見失敗原因**：
1. **預算超出**：OpenAI API 成本超過預算限制
2. **OpenAI API 錯誤**：速率限制、API Key 無效
3. **資料庫錯誤**：連接超時、欄位缺失
4. **回測 API 錯誤**：RAG 服務不可用

**處理步驟**：
1. 查看迴圈詳情頁面的「執行日誌」Tab
2. 查看最近的錯誤事件（event_type = "iteration_failed"）
3. 根據錯誤訊息處理：
   - **預算超出**：增加預算限制或減少批次大小
   - **OpenAI API 錯誤**：檢查 API Key、等待速率限制恢復
   - **資料庫錯誤**：聯繫技術支援
   - **回測 API 錯誤**：確認 RAG 服務是否正常

---

### Q8: 如何暫停或取消迴圈？

**暫停迴圈**：
1. 進入迴圈詳情頁面
2. 點擊「暫停」按鈕
3. 系統標記迴圈為 `paused`
4. 當前迭代會繼續執行完成，但不會自動執行下一次迭代

**恢復迴圈**：
1. 點擊「恢復」按鈕
2. 系統標記迴圈為 `running`
3. 可繼續執行迭代

**取消迴圈**：
1. 點擊「取消」按鈕
2. 確認對話框：「確定要取消迴圈嗎？當前迭代會立即停止。」
3. 系統標記迴圈為 `cancelled`
4. 當前迭代會立即停止（已生成的知識不會丟失）
