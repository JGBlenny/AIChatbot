# 完整發現流程（Full Discovery Process）

> 適用場景：新功能開發、複雜系統整合、外部依賴重度使用

## 目標

確保設計基於完整、準確的資訊，包括：
- 最新的技術最佳實踐
- 外部依賴的實際能力與限制
- 現有程式碼庫的模式與慣例
- 潛在風險與緩解策略

## 執行步驟

### 第 1 階段：需求理解與分類

1. **閱讀需求文件**：完整閱讀 `requirements.md`
2. **識別複雜度指標**：
   - 是否涉及外部 API/服務？
   - 是否需要新的技術棧？
   - 是否有複雜的狀態管理？
   - 是否涉及資料遷移？
   - 是否有高效能要求？
   - 是否有特殊安全需求？

3. **決定調查深度**：
   - 0-1 個「是」→ 考慮輕量發現流程
   - 2-3 個「是」→ 執行完整發現流程
   - 4+ 個「是」→ 執行完整發現流程 + 額外專家諮詢

### 第 2 階段：外部依賴調查（如適用）

#### 2.1 API / 服務調查

對每個外部 API/服務，執行以下調查：

**基本資訊**：
- [ ] 閱讀官方文件（使用 WebFetch）
- [ ] 確認最新版本與相容性
- [ ] 檢查速率限制、配額
- [ ] 了解定價模型

**功能驗證**：
- [ ] 確認 API 是否支援所需功能
- [ ] 檢查請求/回應格式
- [ ] 確認錯誤處理機制
- [ ] 了解重試策略

**整合考量**：
- [ ] 認證方式（API Key, OAuth, JWT）
- [ ] 是否需要 SDK 或可直接呼叫
- [ ] Webhook 支援（如需要）
- [ ] 測試環境可用性

**風險評估**：
- [ ] 服務穩定性（SLA, 歷史停機紀錄）
- [ ] 社群回饋（GitHub Issues, Stack Overflow）
- [ ] 替代方案評估

**範例調查記錄**（記錄至 research.md）：
```markdown
### 外部 API：OpenAI GPT-4 API

**官方文件**：https://platform.openai.com/docs/api-reference

**版本**：v1 (2024-01-01)

**關鍵發現**：
- 支援 streaming response（適用於即時回饋需求）
- 速率限制：3,500 requests/min（需實作速率限制器）
- 錯誤碼：429 (rate limit), 500 (server error) 需重試機制
- 建議使用官方 SDK：`openai@4.x`

**風險**：
- 成本：每 1M tokens 約 $30（需實作使用量監控）
- 服務穩定性：99.9% SLA，過去 6 個月無重大中斷

**決策**：採用，並實作快取層降低成本
```

#### 2.2 函式庫/框架調查

對每個新引入的函式庫：

- [ ] 確認授權相容性（MIT, Apache, GPL）
- [ ] 檢查維護狀態（最後更新時間、issue 回應速度）
- [ ] 評估套件大小（影響 bundle size）
- [ ] 檢查安全漏洞（npm audit, Snyk）
- [ ] 閱讀遷移指南（如果升級現有版本）

### 第 3 階段：架構模式研究

**目標**：選擇適合的架構模式

1. **候選模式識別**：
   - 根據需求特性，列出 2-3 個候選架構模式
   - 範例：Clean Architecture, Hexagonal, Event-Driven, Microservices

2. **模式研究**（使用 WebSearch）：
   ```
   搜尋關鍵字範例：
   - "[架構模式名稱] best practices 2024"
   - "[架構模式名稱] implementation example [語言/框架]"
   - "[架構模式名稱] vs [替代方案]"
   ```

3. **適用性評估**：
   | 模式 | 優點 | 缺點 | 適用性評分 (1-5) |
   |------|------|------|------------------|
   | A | - | - | 4 |
   | B | - | - | 3 |

4. **決策記錄**（記錄至 research.md）：
   - 選擇的模式
   - 選擇理由
   - 與現有架構的對齊方式

### 第 4 階段：現有程式碼分析

**目標**：確保新設計符合現有慣例

1. **模式識別**（使用 Grep）：
   ```bash
   # 搜尋類似功能的實作
   grep -r "class.*Service" --include="*.ts"
   grep -r "interface.*Repository" --include="*.ts"
   grep -r "async.*handle" --include="*.ts"
   ```

2. **檔案結構分析**：
   - [ ] 識別目錄結構慣例
   - [ ] 確認命名規範
   - [ ] 找出相似元件的位置

3. **整合點分析**：
   - [ ] 識別需要修改的現有檔案
   - [ ] 確認現有介面是否需要擴展
   - [ ] 評估影響範圍

4. **技術債務識別**：
   - [ ] 是否有需要重構的舊程式碼？
   - [ ] 是否有不一致的模式？
   - [ ] 是否有機會在本次一併改善？

### 第 5 階段：效能與規模評估

1. **基準測試研究**（使用 WebSearch）：
   - 搜尋類似系統的效能數據
   - 了解常見瓶頸

2. **容量規劃**：
   - 預期資料量
   - 預期請求量
   - 儲存需求

3. **快取策略**：
   - 識別可快取的資料
   - 選擇快取層級（記憶體、Redis、CDN）
   - 定義快取失效策略

### 第 6 階段：安全性威脅模型

1. **STRIDE 分析**：
   - Spoofing（偽造）
   - Tampering（竄改）
   - Repudiation（否認）
   - Information Disclosure（資訊洩漏）
   - Denial of Service（阻斷服務）
   - Elevation of Privilege（提升權限）

2. **輸入驗證點識別**：
   - API 輸入
   - 使用者上傳
   - 外部資料來源

3. **敏感資料處理**：
   - 識別敏感資料（PII, credentials）
   - 加密策略
   - 存取控制

### 第 7 階段：風險登記

建立風險登記表（記錄至 research.md）：

| 風險 ID | 描述 | 影響 | 機率 | 緩解策略 | 負責人 |
|---------|------|------|------|---------|--------|
| R-001 | 外部 API 超出速率限制 | 高 | 中 | 實作速率限制器 + 快取 | - |
| R-002 | 新函式庫包含安全漏洞 | 高 | 低 | 定期執行 npm audit | - |

## 輸出成果

完成發現流程後，必須產出：

1. **research.md** - 完整的調查記錄
2. **技術決策清單** - 關鍵決策及理由
3. **風險登記表** - 識別的風險與緩解策略
4. **架構草圖** - 初步的系統邊界圖

## 時間預估

- 簡單功能：1-2 小時
- 中等複雜度：3-5 小時
- 高複雜度：1-2 天

## 注意事項

- **不要跳過驗證**：即使對某項技術有信心，仍應查證最新文件
- **記錄來源**：所有調查結果必須附上來源連結
- **持續更新**：在設計過程中，如發現新資訊，應更新 research.md
- **平衡深度**：避免過度調查，聚焦於影響設計決策的關鍵資訊
