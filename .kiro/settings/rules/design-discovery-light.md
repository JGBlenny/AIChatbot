# 輕量發現流程（Light Discovery Process）

> 適用場景：功能擴展、簡單整合、CRUD 操作、UI 改善

## 目標

快速收集必要資訊以支援設計決策，聚焦於：
- 現有程式碼模式與慣例
- 整合點與相依性
- 基本風險識別

## 執行步驟

### 第 1 階段：需求快速審查

1. **閱讀需求文件**：閱讀 `requirements.md`
2. **識別關鍵要素**：
   - 需要修改哪些現有元件？
   - 是否引入新的外部依賴？
   - 是否有特殊的非功能性需求？

### 第 2 階段：現有程式碼模式分析

**目標**：找出相似功能的實作方式

1. **搜尋類似功能**（使用 Grep）：
   ```bash
   # 範例：如果要新增 UserService
   grep -r "class.*Service" --include="*.ts"

   # 範例：如果要新增 Repository
   grep -r "interface.*Repository" --include="*.ts"
   ```

2. **識別慣例**：
   - 檔案命名規範（如 `user.service.ts`, `UserService.ts`）
   - 目錄結構（如 `src/services/`, `src/domain/services/`）
   - 介面定義位置（如 `src/interfaces/`, 同檔案內）

3. **提取範例**（記錄至 research.md）：
   ```markdown
   ### 現有模式：Service Layer

   **檔案位置**：`src/services/vendor.service.ts`

   **模式**：
   - 使用 class-based service
   - 依賴注入透過 constructor
   - 錯誤處理使用自訂 Error class
   - 資料驗證使用 Zod schema

   **範例程式碼片段**：
   \```typescript
   export class VendorService {
     constructor(private db: Database) {}

     async getVendor(id: string): Promise<Vendor> {
       // implementation
     }
   }
   \```
   ```

### 第 3 階段：整合點識別

1. **尋找整合點**：
   - 需要呼叫哪些現有 API/Service？
   - 需要修改哪些資料模型？
   - 需要新增哪些資料庫欄位/表格？

2. **檢查介面**（使用 Read）：
   ```typescript
   // 閱讀現有介面定義
   // 確認是否需要擴展或新增
   ```

3. **評估影響範圍**：
   - 低影響：僅新增功能，不修改現有邏輯
   - 中影響：需要擴展現有介面
   - 高影響：需要重構現有邏輯（→ 考慮使用完整發現流程）

### 第 4 階段：外部依賴快速檢查（如適用）

**僅在引入新依賴時執行**

1. **基本檢查**：
   - [ ] 函式庫是否活躍維護？（GitHub stars, 最後更新時間）
   - [ ] 授權是否相容？（使用 WebSearch）
   - [ ] 是否有重大安全漏洞？（npm audit）

2. **API 文件檢查**（使用 WebFetch）：
   - [ ] 確認 API 支援所需功能
   - [ ] 確認請求/回應格式
   - [ ] 確認錯誤處理方式

**簡化記錄**（記錄至 research.md）：
```markdown
### 外部依賴：[函式庫名稱]

**用途**：[簡述]
**版本**：[版本號]
**文件**：[連結]
**注意事項**：[如速率限制、授權、已知問題]
```

### 第 5 階段：基本風險識別

**快速風險掃描**：

| 風險類別 | 檢查項目 | 結果 |
|---------|---------|------|
| 資料完整性 | 是否涉及資料遷移？ | 是/否 |
| 效能 | 是否有大量資料處理？ | 是/否 |
| 安全性 | 是否處理敏感資料？ | 是/否 |
| 相容性 | 是否影響現有 API？ | 是/否 |

**記錄高風險項目**（記錄至 research.md）：
```markdown
### 風險

| 風險 | 緩解策略 |
|------|---------|
| 新增欄位可能破壞現有查詢 | 使用資料庫遷移腳本 + 測試 |
```

## 輸出成果

完成輕量發現流程後，必須產出：

1. **research.md**（簡化版）：
   - 現有模式摘要
   - 整合點清單
   - 基本風險登記

2. **設計草圖**：
   - 簡單的元件關係圖（可選）

## 時間預估

- 簡單功能：15-30 分鐘
- 中等複雜度：30-60 分鐘

## 何時升級為完整發現流程

如果在輕量發現過程中發現以下情況，應升級為完整發現流程：

- ❌ 現有程式碼模式不一致，無明確慣例
- ❌ 需要引入多個新的外部依賴
- ❌ 影響範圍評估為「高影響」
- ❌ 發現多個高風險項目
- ❌ 需求文件中有模糊或矛盾之處

## 注意事項

- **不要假設**：即使是簡單功能，也要驗證現有模式
- **快速迭代**：如果 10 分鐘內無法完成，考慮是否需要完整發現流程
- **記錄發現**：即使是輕量調查，也應記錄關鍵發現至 research.md

---

*輕量發現流程適用於低風險、低複雜度的功能，平衡速度與品質。*
