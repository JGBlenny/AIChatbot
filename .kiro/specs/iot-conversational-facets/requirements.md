# 需求規格：iot-conversational-facets（智慧設備領域對話面向擴充）

> 建立時間：2026-07-04
> 階段：requirements-generated
> 語言：zh-TW
> 承接：contract／billing／account 三 spec（面向化路徑定型：母子分類＋資料四件套＋領域檔 builder＋face 貫穿＋secondary_calls＋收斂/身分程式保底）。
> 定位：**IoT 是 B 端基本功能**（使用者定調 2026-07-04），業者角度屬基本盤，非加值客製。
> Ground truth：jgb2 電表真碼盤查為 research 階段第一件事（**未給電與供電控制機制、離線判定條件、度數同步、儲值扣款鏈、台科電/DAE/skywatch 廠商差異**），需求中標注〔待盤查〕者以 research.md 定案為準。

## 簡介

IoT 領域現況只有單發診斷（`services/jgb/iot.py` diagnose_iot：廠商綁定失敗）與零散知識（IoT 智慧設備 5 筆），無母子分類、無多輪對話、無系統脈絡疊加。**現行主要痛點（使用者一手定調，優先於舊工單）：台科電電表、未給電、設備狀況**。輔助素材：客服 IoT 案 20（電表 9／設定 3／門鎖硬體 6／範圍外 2；年代大宗 2021–2023，需以現碼過濾）。

在母分類 `智慧設備` 下擴充 2 個子面向：

1. **電表排障**（診斷樹）——「租客沒電/未給電」→ ground 儲值餘額×供電狀態×連線現值三路判因；「設備離線/未連線」→ 網路/重啟/廠商串接分支；「度數異常/不增加/與 JGB 不符」→ ground 電表現值與讀數同步機制判讀；「設備狀況」高頻查詢（線上/離線/最後回報/度數）。
2. **IoT 設定引導**（輕引導）——電表串接啟用、儲值操作與單價設定、門鎖預設密碼規則、IoT 連結起始日；1–2 輪分流後知識收尾。

門鎖硬體問題（悠遊卡/磁扣/電池/音量/QRCode）**不開面向**：單發知識包，識別後導廠商或客服（沿 B.Bug 不對話化原則）。

擴充原則沿定型路徑：新增面向＝category_config 子分類＋系統脈絡列＋知識掛面向＋對話 config，**引擎零改程式**（secondary_calls/deterministic_id/suppress_head_id 等既有機制直接沿用）；領域決定性邏輯於 `services/jgb/iot.py` 平行擴充 builder。

## 名詞定義

- **子面向／診斷樹／輕引導／G 清單**：同前三 spec 定義。
- **未給電**：租客端無電力供應的現象。可能因：預付儲值餘額耗盡觸發斷電、供電開關/繼電器被關閉、設備離線導致狀態不明〔三者的判定欄位與優先序待盤查〕。
- **供電控制**：系統/廠商對電表供電的開關機制（自動斷電條件、手動復電路徑）〔待盤查〕。
- **離線判定**：電表被判為離線/未連線的條件（最後回報時間閾值等）〔待盤查〕。
- **度數同步**：廠商設備讀數回傳 JGB 的機制與延遲（「電錶數據與 JGB 不同」的機制解釋）〔待盤查〕。
- **儲值扣款鏈**：儲值餘額→用電扣款→餘額不足處置的資料鏈〔待盤查；與帳務域的儲值「金流入帳」分界：入帳屬帳務、扣款用電屬本域〕。
- **廠商差異**：台科電／DAE／skywatch 等在離線判定、供電控制、綁定機制上的差異〔待盤查；台科電為現行主要痛點廠商〕。

## 範圍

### 範圍內
- 母分類 `智慧設備` 下 2 子面向的四件套資料（系統脈絡、對話 config、知識、API 設定引用）。
- `services/jgb/iot.py` 電表決定性 fact-builder 擴充（face 貫穿機制現成；既有 diagnose_iot 零回歸）。
- `meters`／`recharge-accounts` 端點的 client 包裝（jgb2 已上線，AIChatbot 缺 wrapper）。
- 未給電三路判因、離線分支、度數異常判讀的決定性規則（以 jgb2 真碼盤查為準）。
- 既有 IoT 知識（5 筆）盤點補標＋新知識產製（現行痛點為主、舊工單過濾後為輔、help 素材）。
- 門鎖硬體單發知識包（含導廠商/客服出口）。
- IoT G 清單契約的**定義**（缺欄位盤查後定；jgb2 端實作為外部依賴）。
- 進場路由回歸測試擴 IoT 案例（含與帳務/帳號錨點互不誤吸）。

### 範圍外
- 不改對話引擎核心與既有程式保底（全部直接沿用）。
- 不回歸合約 6／帳務 5／帳號 4 面向、售前、既有 form_fill 診斷行為。
- **租客看不到電表屬帳號域登入排障**（已解，共用出口 scope=switch）。
- **電表儲值的帳單/金流入帳屬帳務域**（儲值「錢進來了沒」→帳務；「電怎麼沒來」→本域）。
- 硬體故障（skywatch 螢幕、QRCode 失靈、電池）不對話化——識別後導廠商/客服。
- VR/istaging 範圍外。
- 不做遠端控制操作（開關電/重啟設備）——AI 不代操作，只判因與指路。
- 不負責 jgb2 端 G 清單欄位的程式實作與部署時程。

## 需求

### Requirement 1：面向擴充的資料驅動底座

**使用者故事**：作為維護者，我要以純資料方式在 `智慧設備` 母分類下掛 2 個新子面向，引擎零改程式即可路由、載入脈絡並進入對話。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以 `category_config` 子分類（parent=`智慧設備`）識別每個新面向：`電表排障`、`IoT設定引導`〔面向名由設計定案〕。
2. WHEN 使用者問題命中某面向掛載的知識，THE SYSTEM SHALL 進入該面向的對話 config，行為與前三域進場機制一致。
3. WHEN 面向對話進行中，THE SYSTEM SHALL 注入「通用 base＋母 `智慧設備`＋該子面向」三層系統脈絡，不含其他領域層與售前 append。
4. THE SYSTEM SHALL 使新增/調整面向不需修改引擎/route 程式碼（僅資料與 `services/jgb/` 領域檔）。
5. THE SYSTEM SHALL 使每輪注入的系統脈絡總長度不超過既有上限（4500 字）。
6. THE SYSTEM SHALL 為每個對話 config 使用面向專屬 persona_role 鍵（`pm_iot_*` 系列），避免與既有領域鍵互蓋。

### Requirement 2：電表排障面向（診斷樹）

**使用者故事**：作為管理者，租客說沒電了、或電表看起來怪怪的，我希望系統查這顆電表的實際狀態，告訴我是餘額、供電還是連線的問題、該做什麼。

#### 驗收標準（EARS）
1. WHEN 使用者表達「沒電/未給電」類意圖且電表未鎖定，THE SYSTEM SHALL 追問並以既有候選辨識機制鎖定電表（識別：物件/房號/合約或電表編號〔識別鏈以 research 定案〕）。
2. WHEN 電表已鎖定且現象為未給電，THE SYSTEM SHALL 以現值決定性三路判因：儲值餘額（耗盡→引導儲值與復電機制）、供電狀態（開關/繼電器關閉→復電路徑）、設備離線（狀態不明→離線分支）〔判定欄位與優先序、復電機制以 research 定案〕。
3. WHEN 現象為設備離線/未連線，THE SYSTEM SHALL 依決定性分支作答：〔分支集以 research 定案，至少涵蓋〕網路排查、設備重啟指引、廠商串接狀態（沿 iot.py 綁定診斷）、超過自助範圍導廠商/客服附設備事實。
4. WHEN 現象為度數異常（跳度/不增加/與 JGB 不同），THE SYSTEM SHALL 以電表現值與度數同步機制作答；度數一律引用系統存值，不推估、不重算。
5. WHERE 使用者查詢設備狀況（線上/離線/最後回報/目前度數），THE SYSTEM SHALL 以現值直答（機械解碼）。
6. THE SYSTEM SHALL 於分支說明含機制/時程時，僅使用 research 證實的規則（含台科電等廠商差異），不沿用未證實的客服口徑。

### Requirement 3：IoT 設定引導面向（輕引導）

**使用者故事**：作為管理者，我要啟用電表串接或設定儲值單價，希望系統先確認我的情境再給對的設定路徑。

#### 驗收標準（EARS）
1. WHEN 使用者表達 IoT 設定意圖，THE SYSTEM SHALL 以至多兩輪分流問句確認情境（電表串接/儲值與單價/門鎖密碼規則/連結起始日），再以知識收尾。
2. WHERE 情境涉及特定設備的現況（「這顆為什麼…」），THE SYSTEM SHALL 判 scope=switch 交由電表排障面向接手。
3. WHERE 詢問儲值操作，THE SYSTEM SHALL 區分本域（怎麼儲值/單價/餘額機制）與帳務域（儲值的錢入帳了沒），後者 scope=switch 至帳務。
4. THE SYSTEM SHALL 將串接教學、單價規則等制度 QA 掛於本面向作為單發知識，不強制進對話。

### Requirement 4：門鎖等硬體單發知識包

**使用者故事**：作為管理者，門鎖悠遊卡/磁扣/電池這類硬體問題，我希望一句話得到答案或知道找誰，不需要多輪對話。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 將門鎖硬體類問題（悠遊卡設定、磁扣、電池/接電、音量、QRCode）以單發知識回答，不掛面向、不進對話。
2. WHERE 問題屬硬體故障或廠商功能面，THE SYSTEM SHALL 於知識中明確導向對應廠商或客服（含可提供的設備資訊清單），不臆測修法。
3. THE SYSTEM SHALL 使此類知識與電表排障錨點互不誤吸（路由測試把關）。

### Requirement 5：決定性計算與回應合成

**使用者故事**：作為維護者，我要未給電判因、離線判定、度數判讀全由程式決定性算出，LLM 只轉述，杜絕機制與數字幻覺。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 由 `services/jgb/iot.py` 領域 builder 決定性計算：未給電三路判因、離線分支、度數異常判讀、設備狀況解碼；LLM 不得自行推斷這些結論。
2. THE SYSTEM SHALL 以 API 即時現值為準；度數與金額（儲值餘額/單價）一律引用系統存值欄位，禁止 LLM 計算或改寫數字。
3. THE SYSTEM SHALL 於 G 欄位缺失時降級為通用指引或引導問句，不虛構欄位值、不阻斷對話（存在性驅動）。
4. WHERE 判因需要第二資料源（如電表＋儲值帳戶），THE SYSTEM SHALL 以既有 `secondary_call(s)` 宣告式機制取得，不為單一面向改引擎。
5. THE SYSTEM SHALL 不代執行任何遠端控制操作（復電/重啟）——只判因、給操作路徑或導廠商/客服。

### Requirement 6：面向間路由與切換

**使用者故事**：作為使用者，我從沒電聊到儲值、或牽涉租客登入，希望對話自然接手。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 沿用既有中途切換機制使 `智慧設備` 母下面向可互轉（設定引導→電表排障），已鎖定設備識別保留不重問。
2. WHERE 話題跨域，THE SYSTEM SHALL 以 scope=switch 關會話重路由：租客看不到電表→帳號域登入排障；儲值金流入帳→帳務域。
3. THE SYSTEM SHALL 使口語進場錨點涵蓋現行痛點高頻問法（未給電/沒電/設備離線/度數怪），**target_user=['property_manager']**；教學/制度問句維持單發（進對話 vs 單發準則）。
4. THE SYSTEM SHALL 使 IoT 錨點與帳務（「儲值」語彙）、帳號（「看不到電表」語彙）錨點互不誤吸（雙向驗證，已知風險點點名覆蓋）。

### Requirement 7：知識素材產製

**使用者故事**：作為知識維護者，我要以現行痛點與過濾後的官方素材產製 IoT 知識，過時與錯誤口徑以程式證據修正。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以現行痛點（台科電/未給電/設備狀況）為知識產製主軸；客服 20 舊案經 jgb2 現碼過濾後轉製（已被改版解決者剔除或改寫為現行行為），help 中心 IoT 相關文章為輔。
2. THE SYSTEM SHALL 對既有 IoT 知識（5 筆）逐筆盤點：歸屬主面向（互斥）或維持單發，經人工確認後補標。
3. THE SYSTEM SHALL 於知識內容與 jgb2 真碼衝突時以 research 盤查結果為準，並修正既有錯誤知識。
4. THE SYSTEM SHALL 使知識遵循品質標準：question_summary 短主題關鍵字、answer 先述情境再帶條件、target_user/業態正確標注、個案資訊剝除、全部進審核流程。

### Requirement 8：外部 API 契約（IoT G 清單）

**使用者故事**：作為整合者，我要明確定義電表排障所需的查詢能力與欄位，兩端獨立部署互不阻塞。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 包裝既有 `meters`／`recharge-accounts` 端點為 client 方法並納入 registry（jgb2 已上線，僅消費端缺 wrapper）。
2. THE SYSTEM SHALL 於 research 盤查兩端點欄位後，對「未給電三路判因」缺失的欄位（如供電/繼電器狀態、最後回報時間、斷電原因）產出 IoT G 清單契約文件交付 jgb2。
3. THE SYSTEM SHALL 以存在性驅動消費 G 欄位：欄位出現即自動啟用分支，缺失走降級，兩端無版本協商。
4. WHERE 既有欄位已足，THE SYSTEM SHALL 優先使用，不重複要求擴充。

### Requirement 9：回歸保護

**使用者故事**：作為維護者，我要確保 IoT 擴充不破壞既有三領域與檢索行為。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 使合約 6／帳務 5／帳號 4 面向、售前、既有 form_fill 與 diagnose_iot 之行為不因本擴充而改變（既有測試全綠）。
2. THE SYSTEM SHALL 擴充進場路由回歸測試：IoT 各面向錨點句進對話＋教學/硬體問句維持單發，含與帳務/帳號錨點互不誤吸的雙向驗證。
3. THE SYSTEM SHALL 為每個新面向提供可回歸執行的測試：分支決策（決定性單元測試）、對話流程（整合測試）、進場錨點（e2e 口語第一句，走正常管線含 reranker）。
4. WHEN 面向資料與知識部署，THE SYSTEM SHALL 將 reranker semantic model 重建納入部署步驟（併統一 runbook）。
5. THE SYSTEM SHALL 使跨域切換（IoT↔帳務、IoT↔帳號）之 e2e 各至少一例。
