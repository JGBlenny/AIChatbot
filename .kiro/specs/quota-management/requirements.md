# 需求規格：quota-management（團隊 AI 客服對話額度管制）

> 建立時間：2026-07-06　階段：requirements-generated　語言：zh-TW
> 承接：`usage-metering`（事實層——usage_events 事件、內部流量標記、複合索引、後台使用量統計頁均為現成底座）。
> 決策沿革（與使用者對話定案 2026-07-06）：額度單位＝訊息數／通知管道＝**對話內直接回應**＋後台進度條（不建 email/webhook）／警示閾值**每團隊可設**／達限＝middleware 攔截（雙受眾文案）／未設額度團隊不受限（opt-in）。

## 簡介

在 usage-metering 之上建立團隊（vendor）月額度管制：後台為團隊設定每月訊息額度（如 5,000 則）與警示閾值（如 80%）；接近額度時在**業者用戶的對話回答內**附額度提示、後台進度條轉黃；達限後該團隊請求由系統直接回應加值引導（不再消耗 LLM 成本），租客與業者看到不同文案；加值（調整額度）即時恢復服務。

核心立場：
- **對話即通知管道**——使用者在哪裡消耗額度，就在哪裡收到額度訊息；不新建外部通知設施。
- **受眾隔離**——「請加值」只給付費決策方（b2b 業者用戶）；租客（b2c）永不看到商業訊息。
- **超限零成本**——達限攔截發生在進檢索/LLM 之前。
- **opt-in**——未設定額度的團隊行為與現狀完全一致。

## 名詞定義

- **額度（quota）**：一個 vendor 的單一日曆月（Asia/Taipei，沿 usage-metering 日界）內可用的非內部訊息數上限。
- **本月用量**：usage_events 中該 vendor 當月 `is_internal=false` 事件數（內部流量不計入）。
- **警示閾值（warning threshold）**：百分比；本月用量 ≥ 額度×閾值 時進入警示狀態。
- **警示狀態／達限狀態**：警示＝繼續正常服務＋提示；達限＝攔截回應。
- **加值**：後台調高該團隊本月可用額度（含臨時追加），即時生效。

## 範圍

### 範圍內
- 額度設定資料（每 vendor：月額度、警示閾值、達限攔截開關）＋後台設定 UI。
- 請求時額度檢查（快取化，不得顯著增加延遲）。
- 警示提示（b2b 回答尾端附註）與達限攔截（雙受眾文案）。
- 後台使用量統計頁：每團隊額度進度條（正常/警示/達限三色）。
- 額度重置（自然月）與加值生效機制。
- 部署併統一 runbook。

### 範圍外
- 金流、收款、自動扣款、發票（加值的商務流程走線下，本 spec 只管額度數字）。
- email／簡訊／webhook 通知（決策：不建）。
- token 成本額度或對話數額度（單位先做訊息數；欄位設計不排除未來擴充）。
- jgb2 側改動。

## 需求

### Requirement 1：額度設定

**使用者故事**：作為 JGB 營運方，我要為每個團隊設定每月對話額度與警示閾值，作為服務方案的一部分。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 支援每 vendor 一筆額度設定：每月訊息額度（正整數）、警示閾值（1–99 的百分比，預設 80）、達限攔截開關（預設開）、啟用狀態。
2. WHERE vendor 無額度設定或設定停用，THE SYSTEM SHALL 不做任何額度檢查與提示（行為與現狀完全一致，opt-in）。
3. THE SYSTEM SHALL 提供後台設定介面（沿既有 admin 認證與版面慣例）：檢視/新增/調整各團隊額度與閾值。
4. WHEN 營運方調整額度（含加值追加），THE SYSTEM SHALL 使新額度於下一次請求檢查即生效（不需重啟）。
5. THE SYSTEM SHALL 於每自然月（Asia/Taipei）自動重置用量計算基準（額度按月計，無結轉）。

### Requirement 2：用量檢查

**使用者故事**：作為維護者，我要額度檢查對正常請求近零負擔，且與計量體系口徑一致。

#### 驗收標準（EARS）
1. WHEN 有額度設定的 vendor 發出請求，THE SYSTEM SHALL 以本月非內部事件數對照額度判定狀態（正常/警示/達限）。
2. THE SYSTEM SHALL 使內部流量（is_internal=true）不計入用量、其請求也不受攔截（回測/迴圈永不被額度擋）。
3. THE SYSTEM SHALL 使檢查機制具快取（短 TTL），對請求延遲的影響與 usage-metering R1.5 同級（中位數 <5ms）。
4. WHERE 快取造成邊界誤差（達限前後數則內），THE SYSTEM SHALL 以「寬鬆不誤殺」為原則（寧可多放行數則，不得提前攔截）。
5. WHEN 計量總開關（USAGE_METERING_ENABLED）關閉，THE SYSTEM SHALL 一併停用額度檢查（無事實即無管制）。

### Requirement 3：警示提示（快到額度）

> ⚠️ 2026-07-06 使用者改判（實作驗收後）：**警示不進對話、改寄 email**——R3.1 的對話內附註改為
> env `QUOTA_WARN_IN_CHAT` 旗標（預設關）；新增 R3.5：每 vendor 每月首次跨越閾值寄警示信
> （收件：`QUOTA_WARN_EMAIL_TO` 營運信箱＋vendor contact_email；DB 月標記防重；SMTP 未設定僅記 log；
> 通知失敗不影響任何請求）。R3.4 後台進度條不變。攔截（R4）仍於對話內回應。

**使用者故事**：作為業者，我的團隊快用完本月額度時，我要在使用對話的當下就知道，來得及決定加值。

#### 驗收標準（EARS）
1. WHEN 本月用量達警示閾值且未達限，且請求來自 b2b 業者用戶（mode=b2b 且具 role_id），THE SYSTEM SHALL 於正常回答尾端附加額度提示（含已用量/額度/百分比，如「本月 AI 客服額度已使用 82%（4,100/5,000）」）。
2. WHERE 請求來自租客（b2c）或 prospect，THE SYSTEM SHALL 不附加任何額度或商業訊息（正常回答不變）。
3. THE SYSTEM SHALL 使警示提示不影響回答本體內容與既有格式（append-only，不得干擾表單/面向流程的結構化輸出）。
4. THE SYSTEM SHALL 於後台使用量統計頁呈現每團隊額度進度條：正常/警示（達閾值）/達限三種視覺狀態。

### Requirement 4：達限攔截與加值引導

**使用者故事**：作為 JGB 營運方，團隊用爆額度後我不想繼續補貼 LLM 成本，但要讓對方清楚知道怎麼恢復服務。

#### 驗收標準（EARS）
1. WHEN 本月用量達額度且該團隊攔截開關為開，THE SYSTEM SHALL 於進入檢索/LLM 之前直接回應（該次請求不產生 LLM 成本）。
2. WHERE 攔截時請求來自 b2b 業者用戶，THE SYSTEM SHALL 回應加值引導文案（含額度數字與聯繫金箍棒客服加值的指引）。
3. WHERE 攔截時請求來自租客（b2c），THE SYSTEM SHALL 回應中性服務暫停文案（引導聯繫其管理公司或客服專線），不得含「加值」等商業字眼。
4. WHERE 團隊攔截開關為關，THE SYSTEM SHALL 達限後僅持續警示提示（b2b）而不攔截（寬限模式）。
5. WHEN 營運方加值（調高額度）後，THE SYSTEM SHALL 使該團隊下一次請求即恢復正常服務。
6. THE SYSTEM SHALL 將被攔截的請求照常記入 usage_events（status 可辨識為額度攔截），供統計與爭議舉證；攔截事件不計入額度用量。

### Requirement 5：可靠性、回歸與部署

**使用者故事**：作為維護者，額度功能上線不得影響任何未設額度團隊的行為，且故障時寧可放行。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 使未設額度團隊與計量關閉情境下零行為差異（既有測試全綠）。
2. WHERE 額度檢查本身發生錯誤（DB 不可達等），THE SYSTEM SHALL fail-open 放行請求並記 log（額度管制失效優於服務中斷）。
3. THE SYSTEM SHALL 以冪等 migration 建立設定表，併入統一部署 runbook。
4. THE SYSTEM SHALL 為檢查邏輯、狀態判定（正常/警示/達限）、受眾文案分流提供 unit 測試（先紅後綠），並以真請求驗證警示與攔截行為（integration）。
5. THE SYSTEM SHALL 於 `make audit` 增加額度一致性檢查條目〔設計定案：如「有攔截事件的 vendor 必有啟用中的額度設定」〕。

## 開放問題（設計階段定案）

1. 警示閾值單段或多段（80%＋95%？）——先單段可設，欄位不排除擴充。
2. 攔截事件的 status 值域（`quota_blocked`？）與報表呈現方式。
3. 警示提示的附加頻率（每則都附 vs 每 N 則附一次，避免洗版）。
4. 額度進度條放使用量統計頁 vs 獨立額度管理頁（傾向同頁）。
