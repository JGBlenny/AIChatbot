# 需求規格：contract-conversational-facets（合約領域對話面向擴充）

> 建立時間：2026-07-02
> 階段：requirements-generated
> 語言：zh-TW
> 承接：`domain-conversational-facets`（面向化三層系統脈絡）與 `conversational-diagnosis`（合約狀態診斷）之上；本 spec 在母分類 `系統合約` 下新增 5 個子面向。
> Ground truth：見本 spec `research.md`（jgb2 四路盤查，file:line 證據；§六 異動申請書出口；§七 面向收斂定案）。

## 簡介

現況合約領域只有 `狀態判斷` 一個子面向（答「這份合約現在在哪」）。依客服資料頻率分析（A 客戶回饋 349 筆中合約類 137 筆、B 異動單 67 筆中合約收尾類 43%、C-1 QA 官方回覆、help 中心合約相關 45 篇），擴充 5 個子面向，按價值×依賴排序：

1. **合約異動**（診斷樹）——「我要改合約上的東西」→ 依狀態與權限分流到三出口，含異動申請書收斂產出。
2. **退租收尾**（步驟推進）——「租客要退租了/退完怎麼還沒結束」→ 依旗標判斷卡在哪一步，給個人化下一步。
3. **續約**（輕引導）——「快到期了怎麼續」→ ground 到期日與續約鏈，講清楚取捨。
4. **建約引導**（輕引導）——「怎麼建約/欄位怎麼填」→ 1–2 輪分流後知識收尾，不需 grounding。
5. **簽署排障**（診斷樹）——「租客簽不了/看不到合約」→ 依簽署旗標與發送通道分支排查。

擴充原則沿 `facet-architecture.md` §六 資料驅動路徑：新增面向＝category_config 子分類＋系統脈絡母/子列＋知識掛面向＋（診斷型）對話 config，**引擎零改程式**；僅 jgb2 外部 API 需小幅擴充（G1–G4，見 research.md §四）。

## 名詞定義

- **子面向（facet）**：母分類 `系統合約` 下的一個 `category_config` 子分類，對應一組系統脈絡列＋知識＋（診斷型）對話 config。
- **診斷樹**：多輪對話面向，收斂時以 API grounding 鎖定單一合約後按決定性規則分支。
- **輕引導**：1–2 輪分流問句後以知識收尾的對話面向，收斂不依賴或僅輕度依賴 API。
- **三出口**：合約異動的分流結果——①待發送直接改 ②待簽取消退回再改 ③已簽走複製重建或資料異動申請書。
- **異動申請書出口**：對話收斂異動資訊後，產出可抄錄的申請內容＋官方範本與提交指引（research.md §六）。
- **G1–G4**：jgb2 外部 API 待擴充欄位（簽署時間戳、登入信箱、帳單封存、is_newest）。
- **決定性計算**：狀態/旗標解碼與「可否操作」由 formatter/API mapping 以程式邏輯算出，LLM 僅把 facts 轉述為口語。

## 範圍

### 範圍內
- 母分類 `系統合約` 下新增 5 個子面向的四件套資料（系統脈絡、對話規則 config、知識、API 設定引用）。
- 合約異動的三出口決策與異動申請書收斂產出。
- 退租收尾的步驟推進與帳單封存判斷（G3 降級策略）。
- 簽署排障的現況分支（誰沒簽/發送通道/綁定）與 G1/G2 上線後的完整分支。
- 續約與建約引導的輕引導行為。
- formatter/mapping 的決定性規則擴充（新面向所需欄位集）。
- 知識素材產製：help 中心兩系列（公開 API 抓取）＋客服 Excel 案例，照知識品質標準轉製並掛面向。
- jgb2 外部 API 擴充需求的**定義**（G1–G4 欄位契約與降級行為）；jgb2 端實作為外部依賴。

### 範圍外
- 不改對話引擎核心（prepare/ask/converge 範式、候選辨識、中途切換機制）。
- 不改檢索 pipeline、reranker、表單機制、既有 DB schema 核心。
- 不回歸既有 `狀態判斷` 面向與售前對話行為。
- 不做特殊建檔個案（包租 A/B/C/D 等）與系統事故的對話化——識別後導客服。
- 不做線上異動表單或異動 API——系統無此模組，出口為申請書文件流。
- 不負責 jgb2 端 G1–G4 的程式實作與部署時程。
- 帳單、社區、物件、IoT 領域（後續 spec）。

## 需求

### Requirement 1：面向擴充的資料驅動底座

**使用者故事**：作為維護者，我要以純資料方式在 `系統合約` 母分類下掛新子面向，讓引擎零改程式即可路由、載入脈絡並進入對話。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以 `category_config` 子分類（parent=`系統合約`）識別每個新面向：`合約異動`、`退租收尾`、`續約`、`建約引導`、`簽署排障`。
2. WHEN 使用者問題命中某面向掛載的知識（config_for_category 命中），THE SYSTEM SHALL 進入該面向的對話 config，行為與既有 `狀態判斷` 進場機制一致。
3. WHEN 面向對話進行中，THE SYSTEM SHALL 注入「通用 base＋母 `系統合約` 共用＋該子面向」三層系統脈絡，且不含售前 append。
4. THE SYSTEM SHALL 使新增/調整面向不需修改 `conversational_engine`、`system_context`、`chat.py` 路由之程式碼（僅資料與設定）。
5. THE SYSTEM SHALL 使每輪注入的系統脈絡總長度不超過既有上限（4500 字）。

### Requirement 2：合約異動面向（診斷樹＋三出口）

**使用者故事**：作為管理者，我說「我想改合約的租期/租金/承租人」，我希望系統告訴我這份合約現在能不能改、該走哪條路，而不是丟一篇通用教學給我。

#### 驗收標準（EARS）
1. WHEN 使用者表達修改合約意圖且未鎖定合約，THE SYSTEM SHALL 追問並以既有候選辨識機制鎖定單一合約（grounding：status-overview，現有欄位即可）。
2. WHEN 合約已鎖定，THE SYSTEM SHALL 依 `status` 決定性分流三出口：status=1 →「可直接編輯」並給操作路徑；status∈{2,4} →「取消退回待發送再改」並說明主取消路徑會保留租客資料；status≥8 →「不可直接改」，分流至複製重建或異動申請書。
3. WHERE 使用者反映「編輯不了合約」，THE SYSTEM SHALL 先分流「狀態擋 vs 權限擋」：引導確認角色是否僅查閱權限，再進狀態分支。
4. WHEN 走異動申請書出口，THE SYSTEM SHALL 收斂異動資訊（異動項目類別、合約 ID、異動前→異動後描述），產出可直接抄錄至申請書的文字，並附範本取得方式、用印與提交指引（會員信箱寄 service@jgbsmart.com 或紙本郵寄）與「金箍棒保留異動與否及費用權利」提醒。
5. WHERE 使用者為團隊管理者，THE SYSTEM SHALL 於申請書指引中提示會員帳號應為團隊擁有者。
6. THE SYSTEM SHALL 於已簽分支說明「藍字（簽後印上 PDF 之參數）不可改、DB 資料可經申請調整、二者實際收款不一致之風險」之區分。
7. WHERE 異動訴求屬「轉歷史合約/刪除」，THE SYSTEM SHALL 轉入共用出口：說明到期自動轉歷史時點；未簽合約引導取消/刪除；已簽合約走異動申請書。

### Requirement 3：退租收尾面向（步驟推進）

**使用者故事**：作為管理者，租客提前解約或到期退租，我希望系統依這份合約的實際進度告訴我「你現在卡在哪、下一步做什麼」，直到收尾完成。

#### 驗收標準（EARS）
1. WHEN 使用者表達退租/提前解約/點退相關意圖且合約已鎖定，THE SYSTEM SHALL 以 bit_status 旗標（256/512 提前解約、64/128 點退、16/32 點交）決定性判斷目前步驟，並給出下一步操作指引。
2. THE SYSTEM SHALL 維持點交與點退互不相依的既有 formatter 規則（可不點交直接點退）。
3. WHEN 收尾步驟到「剩餘帳單封存」且帳單封存欄位（G3）已可用，THE SYSTEM SHALL 查詢該合約帳單封存狀態並個人化指出未封存帳單；WHERE G3 尚未上線，THE SYSTEM SHALL 給出通用封存指引（帳單總表搜合約 ID 批次封存）而不阻斷對話。
4. WHERE 使用者問「點退完成了為什麼還是執行中/什麼時候變歷史合約」，THE SYSTEM SHALL 說明每日排程轉歷史之時點（過終止/到期日後）；WHERE 已逾期仍未轉，THE SYSTEM SHALL 導客服盤查。
5. WHERE 提前解約尚未發起，THE SYSTEM SHALL 說明發起路徑與回簽效期（預設 30 天），並提醒提前產出的帳單需封存。

### Requirement 4：續約面向（輕引導）

**使用者故事**：作為管理者，合約快到期，我希望系統依我這份合約的狀況說明系統續約與重建新約的取捨，以及續約後在哪裡看到新約。

#### 驗收標準（EARS）
1. WHEN 使用者表達續約意圖且合約已鎖定，THE SYSTEM SHALL 以 `date_end` 說明剩餘天數，並以「原約需雙簽完成且未過期」為前提判斷可否走系統續約。
2. THE SYSTEM SHALL 說明系統續約 vs 重建新約之取捨，包含：日期以外參數（租金、電費等）變更需重新簽訂、申請租屋補助時建議重建新約。
3. WHERE 租客為免註冊，THE SYSTEM SHALL 說明續約由管理者單方確認生效、不需租客重簽；WHERE 租客已註冊，THE SYSTEM SHALL 說明需雙方重簽（邀請效期 72 小時）。
4. WHERE 使用者反映「已達數量上限」，THE SYSTEM SHALL 以訂閱方案合約配額（含歷史約與每筆續約皆計數）解釋真因，不得沿用「物件下架」之錯誤說法。
5. THE SYSTEM SHALL 說明續約後列表僅顯示最新一筆、原合約以分頁呈現續約增補文件之行為。

### Requirement 5：建約引導面向（輕引導、無 grounding）

**使用者故事**：作為管理者，我要建立一份合約，我希望系統先問清楚我的情境（全新簽/已有紙本/對象型態），再給對的路徑，並在簽出去前提醒我改不了的參數。

#### 驗收標準（EARS）
1. WHEN 使用者表達建約意圖，THE SYSTEM SHALL 以至多兩輪分流問句確認情境（建約方式：電子簽新約/紙本上傳/舊約要改/續約；對象型態：個人/法人/多人/商用店面），再以知識收尾。
2. THE SYSTEM SHALL 於收尾回答中包含「簽出去前把不可改參數設定正確」之提醒（電價、租金等藍字簽後不可改）。
3. WHERE 情境為「舊約內容要改」，THE SYSTEM SHALL 轉入合約異動面向（R2），不重複引導。
4. WHERE 情境為續約，THE SYSTEM SHALL 轉入續約面向（R4）。
5. THE SYSTEM SHALL 將共同承租話術與條款法律知識（滯納金/審閱期/身分證欄位/商用店面/歷史合約保存年限）掛於本面向作為單發知識，不另開面向。
6. WHERE 訴求屬特殊建檔個案（如包租多戶租補建檔），THE SYSTEM SHALL 識別並導客服，不嘗試對話化解答。

### Requirement 6：簽署排障面向（診斷樹）

**使用者故事**：作為管理者，租客說簽不了約或看不到合約，我希望系統查這份合約實際卡在哪（誰沒簽、發到哪個信箱、效期過了沒），給我能直接執行的解法。

#### 驗收標準（EARS）
1. WHEN 使用者表達簽署問題且合約已鎖定，THE SYSTEM SHALL 以 bit_status 決定性判斷簽署進度（bit 4 租客已簽、bit 8 管理者已回簽），指出「還差誰簽」。
2. THE SYSTEM SHALL 以 `to_user_email`/`to_user_phone` 說明邀請發送通道，並以 `to_user_connect` 判斷租客是否已綁定會員帳號。
3. WHERE G2（租客登入信箱）已可用，THE SYSTEM SHALL 決定性比對發送信箱與登入信箱並於不一致時指出錯配（SSO 情境）；WHERE G2 未上線，THE SYSTEM SHALL 以引導問句協助使用者自行比對。
4. WHERE G1（簽署時間戳/效期）已可用，THE SYSTEM SHALL 判斷邀請是否已過效期，並說明過期會自動退回待發送且清空租客資料、需重新發送。
5. WHERE 排查分支無法以現有資料判定（如簽名格層級、簡訊未達、電子簽供應商延遲），THE SYSTEM SHALL 給出標準自助排查步驟（瀏覽器/驗證碼/簡訊攔截）後收斂導客服。

### Requirement 7：決定性計算與回應合成

**使用者故事**：作為維護者，我要所有狀態解碼與「可否操作」判斷由程式決定性算出，LLM 只負責把結果講成人話，避免幻覺斷言。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 由 formatter/API mapping 決定性計算：三出口分流、退租步驟判定、簽署進度、續約可行性；LLM 不得自行推斷這些結論。
2. THE SYSTEM SHALL 以 API 即時現值為準，面向系統脈絡僅供解讀，不得以通則覆寫該筆合約實際資料。
3. WHEN API 查無或使用者指涉不明（「剛剛那份」無法對應），THE SYSTEM SHALL 依既有規則請使用者補識別，不憑印象斷言。
4. THE SYSTEM SHALL 於 grounding 欄位缺失（G1–G4 未上線）時降級為通用指引或引導問句，不得虛構欄位值、不得阻斷對話。

### Requirement 8：面向間路由與切換

**使用者故事**：作為使用者，我在問狀態時順口問「那我可以改租金嗎」，我希望對話自然接到異動面向，不需要重新開始。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 沿用既有中途切換機制使 6 個合約子面向（含既有 `狀態判斷`）間可互轉，切換後注入目標面向之系統脈絡。
2. WHEN 面向互轉且合約已於前面向鎖定，THE SYSTEM SHALL 保留已鎖定之合約識別，不重複要求識別。
3. THE SYSTEM SHALL 使口語進場錨點涵蓋各面向高頻問法（自客服 Excel 真實問句取材），含「合約」口語第一句可進場之既有保證不回歸。

### Requirement 9：知識素材產製

**使用者故事**：作為知識維護者，我要以官方口徑為主幹、客服個案為邊界補充，產出各面向的知識與系統脈絡內容，且全部經人工審核。

#### 驗收標準（EARS）
1. THE SYSTEM（產製流程）SHALL 以 help 中心公開 API（getGroups/post）抓取「新手上路」「常見問題」兩系列合約相關文章作為官方口徑素材，以客服 Excel（A/C-1）案例作為真實問法與邊界素材。
2. THE SYSTEM SHALL 依既有知識品質標準產製：question_summary 用短主題關鍵字、answer 先述情境再帶條件、不模板化、不直接搬運原文。
3. THE SYSTEM SHALL 將產製知識掛至對應子面向分類，並依 target_user（property_manager 為主、租客題掛 tenant）與業態設定隔離。
4. WHERE 客服摘要與 jgb2 程式盤查結論不一致（續約配額真因、免註冊不重簽、取消保留租客資料之路徑差異），THE SYSTEM SHALL 以程式盤查版本為準。
5. THE SYSTEM SHALL 使所有新知識與系統脈絡列經人工審核後方生效。

### Requirement 10：外部 API 擴充契約（G1–G4）

**使用者故事**：作為整合方，我要明確定義 jgb2 外部 API 需補的欄位與未上線時的系統行為，使兩端可獨立部署。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 定義 G1–G4 欄位契約：status-overview 加 `contract_inviting_at`/`contract_inviting_expire_at`/`contract_inviting_sign_at`/`contract_finish_sign_at`（G1）、`to_user_login_email`（G2）、`is_newest`（G4）；bills 加 `is_archived`/`archive_ymd`（G3）。
2. WHILE 任一 G 欄位未上線，THE SYSTEM SHALL 依 R7.4 降級運作，且該面向其餘分支不受影響。
3. WHEN G 欄位上線，THE SYSTEM SHALL 不需改程式即可啟用對應分支（欄位存在性驅動）。

### Requirement 11：回歸保護

**使用者故事**：作為維護者，我要確保擴充不破壞既有面向與檢索基準。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 使既有 `狀態判斷` 面向與售前對話之行為、系統脈絡內容不因本擴充而改變（既有測試全綠）。
2. THE SYSTEM SHALL 使知識檢索既有基準（80% 命中率基準集）不因新增知識而回歸。
3. THE SYSTEM SHALL 為每個新面向提供可回歸執行的測試：分流決策（決定性單元測試）、對話流程（整合測試）、進場錨點（e2e 口語第一句）。
4. THE SYSTEM SHALL 使 e2e 測試走**正常對話處理流程**（意圖分類 → 檢索 → reranker 重排 → config_for_category 路由 → 對話引擎），不得以直呼引擎替代進場與收斂後重路由之驗證。
5. WHEN 會話結束（scope=switch 重路由／關閉／超時），THE SYSTEM SHALL 使後續問句回到正常流程處理（含 reranker），且 e2e 測試涵蓋此銜接；收斂後會話續存供同面向追問（指涉/換識別），維持既有行為（2026-07-02 設計審查定案）。
6. WHEN 新增面向錨點知識與系統脈絡列部署，THE SYSTEM SHALL 將 reranker semantic model 重建納入部署步驟，避免排序與新資料不同步。
