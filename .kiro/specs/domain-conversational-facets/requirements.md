# 需求規格：domain-conversational-facets（領域化對話面向）

> 建立時間：2026-07-01
> 階段：requirements-generated
> 語言：zh-TW
> 承接：`conversational-diagnosis`（對話式合約診斷框架）之上；本 spec 補其真 API 驗證揭露的三缺口。

## 簡介

把「對話式回答/診斷」的面向從「**全域共用一份系統脈絡 + 收斂時各自臨時檢索**」升級為「**以母分類（category_config 父/子分類）組織的領域**」。每個領域（售前、合約、帳單、修繕…）自帶**四件套**：①系統脈絡（該領域的**基底架構知識**，固定注入）②對話規則（persona）③一般知識（領域 FAQ）④（診斷型）API 設定。

三項核心能力：
1. **per-領域系統脈絡**：`get_system_context` 由全域單例改為依領域載入，解決「合約診斷硬吃到售前系統脈絡」。
2. **混合 grounding（R4.1 修訂）**：診斷型收斂時 grounding =「領域基底架構知識」＋「API 即時資料」，讓 LLM 有領域框架可**判斷**（解讀狀態、推斷下一步、依規則回答），而非只複述 API 數字。
3. **母分類組織 + 候選可辨識**：以領域組織資源；並修正「同名候選不可辨識」。

**動機（真 jgb2 preview API 驗證，role_id=20151 回 50 筆真合約）揭露三缺口**：
- 診斷收斂只 grounding 於 API、無領域知識 → LLM 只能複述。
- 系統脈絡全域一份（售前導向）→ 合約診斷吃到售前脈絡。
- 候選同名不可辨識（「基隆溫馨一人宅套房」真回 3 筆同名，label 只有 title）。

**首案＝合約**：其基底架構知識已從真實 jgb2 程式盤查整理（見 research.md，含 file:line）。

## 名詞定義

- **領域（domain / facet）**：一組對話面向資源的集合，以一個**母分類**識別（如「合約」對應 `合約管理`，售前對應其角色範圍）。
- **母分類（parent category）**：`category_config` 中的父分類值，其子分類自動涵蓋（沿用既有父層展開）。
- **四件套**：系統脈絡、對話規則、一般知識、API 設定。
- **系統脈絡（system context / system_md）**：每輪固定注入 LLM 的背景知識文件（`category='系統脈絡'`）；本 spec 使其可分領域。
- **基底架構知識**：某領域穩定的結構/規則知識（合約＝12 狀態/生命週期/違約金滯納金規則），作為該領域系統脈絡內容。
- **混合 grounding**：收斂時同時以「領域知識（穩定）」與「API 回傳（即時現值）」作為 LLM 合成底稿。
- **領域鍵（domain key）**：載入某領域系統脈絡/知識時的鍵，預設沿用 `target_user`（比照既有 `對話規則` 載入），或母分類值。

## 範圍

### 範圍內
- 系統脈絡**可分領域**載入（per-領域），售前續用其原有那份。
- 診斷型收斂改為**混合 grounding**（領域知識 + API），並讓「以何者為準」明確（現值優先、知識供解讀）。
- 以**母分類**組織領域的四件套；診斷面向依 `topic_scope.category` 對應其領域。
- **候選可辨識**：多筆候選的顯示標籤帶區別資訊（設定化）；候選數過多時的處置。
- 交付**合約**為首個領域：其基底架構知識（系統脈絡內容）+ 混合 grounding + 候選辨識，端到端可運作（真 jgb2 API）。
- 通用底座：上述一律讀設定/資料，新領域零改程式。

### 範圍外
- 不修改派發器結構、檢索 pipeline、reranker、表單機制、DB schema 核心。
- 不改既有售前（prospect）對話的進入方式、追問、收斂**行為**（僅其系統脈絡來源改為「領域化」機制下的售前那份，內容不變）。
- 不重做 `conversational-diagnosis` 已完成部分（select:api、插點 A/B、分類路由、種子）；本 spec 為其上之增修。
- 不負責 jgb2 後端 API 本身；不負責合約基底架構知識的最終業務審核（交付草稿，需領域專家覆核）。

## 需求

### Requirement 1：以母分類組織的領域

**使用者故事**：作為維護者，我希望一個對話面向的所有資源（脈絡/規則/知識/API）都掛在同一個「領域」下，這樣新增或調整一個領域時邊界清楚、不互相污染。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以一個**母分類**識別一個領域，並以既有 `category_config` 父/子分類關係涵蓋該領域的子分類（沿用父層展開，不新增機制）。
2. THE SYSTEM SHALL 使一個領域可掛載四件套（系統脈絡、對話規則、一般知識、API 設定），且四者以該領域的**領域鍵**（預設 `target_user`）或母分類彼此關聯。
3. WHERE 一個診斷型對話面向已設定 `topic_scope.category`，THE SYSTEM SHALL 以該分類對應到其所屬領域，據此載入該領域的系統脈絡與知識。
4. THE SYSTEM SHALL 不要求為新領域修改引擎程式；新增領域＝新增母分類 + 四件套資料。

### Requirement 2：per-領域系統脈絡

**使用者故事**：作為維護者，我要讓合約診斷用「合約的系統脈絡」、售前用「售前的系統脈絡」，而不是全部共用一份售前導向的脈絡。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 依**領域鍵**載入對應領域的系統脈絡（system_md），取代「全域單例只載入單一 `系統脈絡` 文件」。
2. WHEN 對話面向為合約診斷，THE SYSTEM SHALL 注入「合約領域的系統脈絡（基底架構知識）」，而非售前系統脈絡。
3. WHERE 某領域未提供專屬系統脈絡，THE SYSTEM SHALL 回退到既有的預設/最小合規系統脈絡（不阻斷、不報錯）。
4. THE SYSTEM SHALL 維持售前（prospect）注入的系統脈絡內容與現況一致（售前不回歸）。
5. THE SYSTEM SHALL 沿用既有快取機制，且快取鍵能區分不同領域（避免跨領域取到錯的脈絡）。

### Requirement 3：混合 grounding（API ＋ 領域知識）— R4.1 修訂

**使用者故事**：作為使用者，我問「我這份合約現在什麼狀態、接下來會怎樣」，我希望系統不只念出狀態，而是能依合約制度解讀我現在在哪一步、下一步是什麼。

#### 驗收標準（EARS）
1. WHEN 診斷型面向收斂且 grounding 方式含 API，THE SYSTEM SHALL 以「領域基底架構知識」與「API 即時回傳」**兩者**作為 LLM 合成的事實底稿（混合 grounding），取代僅以 API 為底稿。
2. THE SYSTEM SHALL 以 API 回傳的**即時現值為準**，領域基底架構知識僅供**解讀/判斷**，不得以通則覆寫該筆合約的實際資料。
3. THE SYSTEM SHALL 由**設定**決定領域知識來源（如母分類或明列知識），不得將特定領域的知識來源寫死於程式。
4. WHERE 某面向未設定領域知識來源，THE SYSTEM SHALL 退回既有「僅 API grounding」行為（向後相容）。
5. THE SYSTEM SHALL 沿用既有 grounding 撈取與合成元件（如既有的分類/ids 撈知識、售前合成路徑），不另寫新的檢索或合成邏輯。

### Requirement 4：多筆候選可辨識與過多候選處置

**使用者故事**：作為使用者，當我的識別對到多份合約時，我希望候選清單能讓我分得出哪一份；如果太多，我希望系統請我補更明確的條件，而不是丟一長串一模一樣的名稱。

#### 驗收標準（EARS）
1. WHEN API 回多筆候選，THE SYSTEM SHALL 以**帶區別資訊**的標籤呈現候選（不僅顯示名稱），區別欄位由**設定**指定（如期間、狀態）。
2. THE SYSTEM SHALL 使候選的區別欄位讀自設定（不寫死特定領域欄位），支援不同領域各自指定。
3. IF 候選數超過設定上限，THEN THE SYSTEM SHALL 不逐一全列，改請使用者補更明確的識別條件（並可提示已省略），避免不可用的長清單。
4. THE SYSTEM SHALL 維持既有「以序號/名稱/id 確定性比對選擇」之機制對辨識後的候選有效。

### Requirement 5：合約首案交付（基底架構知識）

**使用者故事**：作為驗收者，我要看到合約領域真的用上了從程式盤查來的基底架構知識，且對真實 API 跑得通。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 交付「合約領域系統脈絡（基底架構知識）」內容，涵蓋：狀態模型（`status` 單值 vs `bit_status` 位元遮罩）、12 里程碑與 4 大階段、各階段可否動作之前提、提前解約違約金型態（按月數/固定金額/其他）、滯納金規則、續約父子鏈與同名多份合約之處置。
2. THE SYSTEM SHALL 使該基底架構知識與真實 jgb2 程式一致（以 research.md 盤查為據），標示與先前 mock 假設之差異（如點交=交屋/點退=退租、penalty_type 三值）。
3. WHEN 使用者查某一份合約狀態，THE SYSTEM SHALL 以「合約基底架構（解讀）＋合約 API（現值）」合成回覆，能正確解讀 `bit_status`（逐位元，不當單一整數）。
4. THE SYSTEM SHALL 使一般合約知識查詢（如違約責任通則）維持既有靜態知識回答（三出口不變）。

### Requirement 6：通用性（讀設定、零改程式擴充）

**使用者故事**：作為維護者，我希望這次為合約做的領域底座，未來帳單/繳費/修繕等領域只要加母分類 + 四件套資料就能用。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 將領域鍵、系統脈絡來源、領域知識來源、候選區別欄位全部由**設定/資料**驅動，程式中不得出現綁定特定領域（如合約）的硬編值。
2. WHERE 新增另一個領域（如帳單）僅提供「母分類 + 四件套資料」，THE SYSTEM SHALL 無需修改引擎程式即可啟用該領域的 per-領域脈絡與混合 grounding。
3. THE SYSTEM SHALL 使混合 grounding 的知識來源可由設定指向不同領域的母分類/知識。

### Requirement 7：行為保持與重用（不回歸）

**使用者故事**：作為維護者，我要確保這個升級不弄壞既有售前對話、既有合約診斷、既有表單與一般查詢。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 維持既有售前（prospect）對話流程的進入、追問、收斂行為與注入脈絡內容不變。
2. THE SYSTEM SHALL 維持 `conversational-diagnosis` 既有能力（select:api 三路、插點 A/B、分類路由、依分類查設定）之對外行為。
3. THE SYSTEM SHALL 維持非對話面向查詢路徑（既有檢索→表單／直接知識）行為不變。
4. THE SYSTEM SHALL 不變更 DB schema 核心、派發器結構、檢索 pipeline 與後端設定 payload 契約。
5. WHEN 某領域未設定新能力（per-領域脈絡/混合 grounding/候選區別），THE SYSTEM SHALL 以既有預設行為運作（向後相容）。

### Requirement 8：端到端交付與驗證

**使用者故事**：作為驗收者，我要看到合約領域對真實 jgb2 API 完整跑通「解讀型」對話，且既有流程不回歸。

#### 驗收標準（EARS）
1. WHEN 以真實 role_id（測試 20151）對真實 jgb2 合約 API 進行端到端對話，THE SYSTEM SHALL 完成「模糊→追問→（多筆）帶區別候選→選擇→單筆收斂→以基底架構解讀真實狀態」。
2. WHEN 使用者問一般合約知識，THE SYSTEM SHALL 轉靜態知識回答（三出口）。
3. THE SYSTEM SHALL 在候選過多情境改為請補更明確條件（不列長清單）。
4. THE SYSTEM SHALL 通過各層自動化測試（unit/integration；e2e 於具真服務+真 API 環境），且既有售前、合約診斷、一般查詢測試不回歸。
