# 需求規格：billing-conversational-facets（帳務領域對話面向擴充）

> 建立時間：2026-07-03
> 階段：requirements-generated
> 語言：zh-TW
> 承接：`contract-conversational-facets`（面向化路徑定型：母子分類＋資料四件套＋FACE_BUILDERS＋face 貫穿＋收斂/身分程式保底）與 `conversational-diagnosis` v1.1（4 個帳務單發診斷引擎）。
> Ground truth：jgb2 帳務真碼盤查為本 spec research 階段第一件事（金流狀態機、對帳排程、發票開立時點、滯納金結算規則），需求中標注〔待盤查〕者以 research.md 定案為準。

## 簡介

帳務領域現況只有單發診斷（v1.1：`diagnose_bill`／繳費紀錄／發票／訂閱，form_fill 觸發）與零散知識（帳單管理 12＋發票開立 10＋付款金流 5 筆），無母子分類、無多輪對話、無系統脈絡疊加。依素材證據（客服 Excel 帳務類官方回覆 11 則：超商繳費狀態不跳轉、虛擬帳號過期/格式/臨櫃匯款、租客看不到帳單、匯款上限、一元帳單測試；合約基準回測非合約失敗樣本：繳費週期選項、發票補開/差額時機、滯納金規則），在母分類 `系統帳務` 下擴充 5 個子面向，按價值×依賴排序：

1. **繳費金流排障**（診斷樹）——「租客說繳了但錢沒進來/狀態沒跳」→ ground 帳單狀態＋繳費紀錄 → 依金流狀態機分支（超商未轉出、待對帳、虛擬帳號過期、匯款上限、跨行方式）→ 解法或導客服。
2. **帳單異常診斷**（診斷樹）——「金額不對/帳單沒產生/租客看不到」→ ground 帳單明細與合約計費設定，按產生規則與計算鏈決定性判因。
3. **發票**（診斷樹）——「發票什麼時候開/沒開到/要補開/差額/作廢」→ ground 發票紀錄與開立設定。
4. **滯納金**（輕引導＋grounding）——「滯納金怎麼算/為什麼收這麼多/可以調嗎」→ ground 合約滯納金設定與該筆帳單結算，按客製版本講規則。
5. **帳單設定引導**（輕引導）——「週期怎麼設/繳費期限/發送日/收款帳戶」→ 1–2 輪分流後知識收尾，不需 grounding。

擴充原則沿面向化定型路徑：新增面向＝category_config 子分類＋系統脈絡母/子列＋知識掛面向＋（診斷型）對話 config，**引擎零改程式**；領域決定性邏輯於 `services/jgb/bills.py`（及 payments/invoices）平行擴充 FACE_BUILDERS；jgb2 外部 API 缺口以存在性驅動 G 清單處理（research 定案）。

## 名詞定義

- **子面向（facet）**：母分類 `系統帳務` 下的一個 `category_config` 子分類，對應一組系統脈絡列＋知識＋（診斷型）對話 config。
- **診斷樹／輕引導／決定性計算**：同 contract-conversational-facets 定義。
- **金流狀態機**：帳單從產生→發送→租客付款→（超商/銀行轉出）→對帳入帳→完成的狀態轉移與時點規則〔待盤查：jgb2 真碼〕。
- **對帳排程**：金流通知與帳單狀態同步的批次時點〔待盤查〕。
- **客製滯納金版本**：標準百分比外的固定金額版/延遲金備註版/階梯式版（既有知識 3531/3532 記載，以真碼覆核）。
- **G 清單（帳務）**：jgb2 外部 API 為本領域需補露出的欄位集，research 盤查後定案；消費端一律存在性驅動（欄位在才啟用分支，缺則降級不虛構）。

## 範圍

### 範圍內
- 母分類 `系統帳務` 下 5 個子面向的四件套資料（系統脈絡、對話 config、知識、API 設定引用）。
- `services/jgb/` 帳務決定性 fact-builder 擴充（bills/payments/invoices 檔內 FACE_BUILDERS，face 貫穿機制現成）。
- 繳費金流排障的分支決策（含超商/虛擬帳號/匯款上限等真實客服場景）與導客服收斂。
- 帳單異常、發票、滯納金的決定性判因規則（以 jgb2 真碼盤查為準）。
- 既有 4 個單發診斷引擎與新面向的關係定案（升級掛面向 vs 保留單發，逐一判定）。
- 既有帳務知識（27+ 筆）盤點補標＋新知識產製（Excel 官方回覆＋help 素材＋回測缺口清單）。
- jgb2 外部 API 帳務缺口的**定義**（G 清單契約與降級行為）；jgb2 端實作為外部依賴。
- 進場路由回歸測試擴充帳務案例（沿 `test_facet_entry_routing_req.py` 模式）。

### 範圍外
- 不改對話引擎核心與既有程式保底（prepare/ask/converge、候選辨識、face 貫穿、收斂槽位/身分保底、secondary_call 機制——全部直接沿用）。
- 不回歸合約 6 面向、既有條件診斷 form_fill、售前對話行為。
- **點退帳單/帳單封存屬合約域退租收尾**（共用出口，不在帳務域重複開面向）；押金結算屬點退清單流程。
- 金流工程事故（渠道大範圍異常、對帳批次故障）不對話化——識別後導客服。
- 不做線上調帳/退款操作 API——涉及金流異動一律導客服或既有後台流程。
- 不負責 jgb2 端 G 清單欄位的程式實作與部署時程。
- 社區、物件、IoT、修繕領域（後續 spec）。

## 需求

### Requirement 1：面向擴充的資料驅動底座

**使用者故事**：作為維護者，我要以純資料方式在 `系統帳務` 母分類下掛 5 個新子面向，引擎零改程式即可路由、載入脈絡並進入對話。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以 `category_config` 子分類（parent=`系統帳務`）識別每個新面向：`繳費金流排障`、`帳單異常`、`發票`、`滯納金`、`帳單設定引導`。
2. WHEN 使用者問題命中某面向掛載的知識（config_for_category 命中），THE SYSTEM SHALL 進入該面向的對話 config，行為與合約面向進場機制一致。
3. WHEN 面向對話進行中，THE SYSTEM SHALL 注入「通用 base＋母 `系統帳務` 共用＋該子面向」三層系統脈絡，且不含售前 append 與合約領域脈絡。
4. THE SYSTEM SHALL 使新增/調整面向不需修改 `conversational_engine`、`system_context`、`chat.py` 路由之程式碼（僅資料與 `services/jgb/` 領域檔）。
5. THE SYSTEM SHALL 使每輪注入的系統脈絡總長度不超過既有上限（4500 字）。
6. THE SYSTEM SHALL 為每個對話 config 使用面向專屬 persona_role 鍵（不與 `property_manager` 及合約面向鍵共用），避免規則互蓋。

### Requirement 2：繳費金流排障面向（診斷樹）

**使用者故事**：作為管理者，租客跟我說錢繳了，但系統帳單狀態沒動，我希望系統查這筆帳單的實際金流進度，告訴我卡在哪一步、要等還是要處理。

#### 驗收標準（EARS）
1. WHEN 使用者表達「繳了未入帳/狀態沒跳」類意圖且未鎖定帳單，THE SYSTEM SHALL 追問並以既有候選辨識機制鎖定單一帳單（識別：帳單編號、合約編號或物件名稱＋期別）。
2. WHEN 帳單已鎖定，THE SYSTEM SHALL 以帳單狀態與繳費紀錄決定性判定金流階段，並依分支作答：〔分支集以 research 盤查定案，至少涵蓋〕超商已繳未轉出（含預期轉出時程）、已轉出待對帳、虛擬帳號已過期（含重發帳單刷新帳號之解法）、疑似匯款金額上限（引導確認）、查無繳費紀錄（引導核對繳款帳號）。
3. WHERE 金流紀錄與帳單狀態不一致且無分支可判，THE SYSTEM SHALL 收斂導客服並附已查得的決定性事實（帳單編號/狀態/最後金流事件），不臆測原因。
4. THE SYSTEM SHALL 於分支說明中包含「預期時程」時，僅使用 research 盤查證實的排程/時點，不得沿用未證實的客服口徑。
5. WHERE 使用者為租客身分情境（由管理者轉述），THE SYSTEM SHALL 以管理者可執行的動作作答（重發帳單、核對帳號、請租客改繳款方式），不指示租客直接操作管理端。

### Requirement 3：帳單異常診斷面向（診斷樹）

**使用者故事**：作為管理者，帳單金額跟我預期不一樣、或該出的帳單沒出現、或租客說看不到帳單，我希望系統依這筆帳單與合約設定告訴我原因。

#### 驗收標準（EARS）
1. WHEN 使用者表達帳單異常意圖且帳單/合約已鎖定，THE SYSTEM SHALL 以帳單明細與合約計費設定決定性判因：金額類（不足月比例、調整品項、押金/違約金併入）、缺漏類（產生時點未到、合約起始日在過去之補產規則、封存）、可見類（角色視角、帳單狀態）〔計算鏈與產生規則以 research 定案〕。
2. WHEN 判因涉及「帳單怎麼算出來的」，THE SYSTEM SHALL 逐項列出系統存值的組成（品項、金額、期間），不得由 LLM 重算或推估金額。
3. WHERE 異常屬封存/點退帳單範疇，THE SYSTEM SHALL 以共用出口銜接合約域退租收尾（scope=switch），不在本面向重複解答。
4. WHERE 租客反映「看不到帳單」，THE SYSTEM SHALL 依決定性檢查給排查（角色切換、帳務頁位置、帳單狀態），無法判定則導客服。

### Requirement 4：發票面向（診斷樹）

**使用者故事**：作為管理者，租客要發票或發票開錯了，我希望系統查這筆帳單的發票紀錄，告訴我開了沒、何時會開、怎麼補開或作廢重開。

#### 驗收標準（EARS）
1. WHEN 使用者表達發票意圖且帳單已鎖定，THE SYSTEM SHALL 以發票紀錄決定性判定狀態（已開立/待開立/開立失敗/已作廢），並依狀態作答〔開立時點與觸發規則以 research 定案〕。
2. WHERE 詢問補開或差額發票，THE SYSTEM SHALL 說明適用時機與操作入口（依證實的後台功能），超出自助範圍導客服。
3. WHERE 發票設定未啟用（合約或團隊層級），THE SYSTEM SHALL 指出設定位置而非診斷失敗。
4. THE SYSTEM SHALL 不得虛構發票號碼、開立日期等未在紀錄中的值。

### Requirement 5：滯納金面向（輕引導＋grounding）

**使用者故事**：作為管理者，租客質疑滯納金金額，我希望系統依這份合約的滯納金設定與該筆帳單的結算紀錄，把規則講清楚。

#### 驗收標準（EARS）
1. WHEN 使用者詢問滯納金且合約/帳單已鎖定，THE SYSTEM SHALL 以合約滯納金設定（啟用與否、費率、緩衝天數、客製版本）與帳單結算值作答；金額一律引用系統存值，不重算。
2. THE SYSTEM SHALL 正確區分客製版本行為（標準百分比/固定金額/延遲金備註/階梯式）〔各版本結算規則以 research 覆核既有知識 3531/3532〕。
3. WHERE 詢問「可不可以不收/調整」，THE SYSTEM SHALL 說明設定層級與調整途徑（依證實功能），涉及個案減免導客服。
4. WHERE 未鎖定合約而問一般規則，THE SYSTEM SHALL 以知識單發作答（不強制進對話）。

### Requirement 6：帳單設定引導面向（輕引導、無 grounding）

**使用者故事**：作為管理者，我要設定收款方式與帳單週期，希望系統先確認我的情境再給對的設定路徑。

#### 驗收標準（EARS）
1. WHEN 使用者表達帳單設定意圖，THE SYSTEM SHALL 以至多兩輪分流問句確認情境（設定對象：收款帳戶/繳費週期/繳費期限與發送日/繳費方式），再以知識收尾。
2. WHERE 情境涉及特定帳單或合約的現況（「這筆為什麼…」），THE SYSTEM SHALL 判 scope=switch 交由對應診斷面向接手。
3. WHERE 涉及收款帳戶新開通驗證（如一元帳單測試流程），THE SYSTEM SHALL 以證實的官方流程作答。
4. THE SYSTEM SHALL 將繳費週期選項、期限/發送日規則等制度 QA 掛於本面向作為單發知識，不強制進對話。

### Requirement 7：決定性計算與回應合成

**使用者故事**：作為維護者，我要所有金流階段、判因與金額組成由程式決定性算出，LLM 只轉述，杜絕金額幻覺。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 由 `services/jgb/` 領域檔 FACE_BUILDERS 決定性計算：金流階段分支、帳單判因、發票狀態、滯納金規則；LLM 不得自行推斷這些結論。
2. THE SYSTEM SHALL 以 API 即時現值為準；涉及金額時僅引用系統存值欄位，禁止 LLM 計算或改寫數字。
3. WHEN API 查無或識別不明，THE SYSTEM SHALL 依既有規則請使用者補識別；收斂槽位保底與身分參數保底行為沿用（未齊轉追問、身分缺誠實降級）。
4. THE SYSTEM SHALL 於 G 清單欄位缺失時降級為通用指引或引導問句，不得虛構欄位值、不得阻斷對話。
5. WHERE 單筆收斂需第二資料源（如帳單＋繳費紀錄），THE SYSTEM SHALL 以既有 `secondary_call` 宣告式機制取得，不為單一面向改引擎。

### Requirement 8：面向間路由與切換

**使用者故事**：作為使用者，我在問帳單時順口問到合約或反過來，希望對話自然接手，不需重新開始。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 沿用既有中途切換機制使 `系統帳務` 母下 5 面向可互轉，切換後注入目標面向脈絡；已鎖定之帳單/合約識別保留不重問。
2. WHERE 話題跨域（帳務↔合約，如點退帳單、續約帳單時點），THE SYSTEM SHALL 以 scope=switch 關會話重路由至對應領域 config。
3. THE SYSTEM SHALL 使口語進場錨點涵蓋各面向高頻問法（自客服 Excel 與回測失敗樣本取材）；具體操作/制度問句維持單發不掛面向（進對話 vs 單發準則）。
4. THE SYSTEM SHALL 定案既有 `條件診斷：帳單`/`條件診斷：發票` form_fill 知識的處置（升級掛新面向或維持），且不使其行為在未定案前回歸。

### Requirement 9：知識素材產製

**使用者故事**：作為知識維護者，我要以官方素材與真實客服案例產製帳務知識，錯誤口徑以程式證據修正。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以客服 Excel 帳務官方回覆（11 則）、help 中心帳務相關文章、回測缺口清單為素材來源，理解情境後轉製（不直接搬運）。
2. THE SYSTEM SHALL 對既有帳務知識（帳單管理/發票開立/付款金流 27+ 筆）逐筆盤點：歸屬主面向（互斥）或維持單發，經人工確認後補標。
3. THE SYSTEM SHALL 於知識內容與 jgb2 真碼衝突時以 research 盤查結果為準，並修正既有錯誤知識。
4. THE SYSTEM SHALL 使知識遵循品質標準：question_summary 短主題關鍵字、answer 先述情境再帶條件、target_user/業態正確標注、全部進審核流程。

### Requirement 10：外部 API 契約（帳務 G 清單）

**使用者故事**：作為整合者，我要明確定義 jgb2 需補露出的帳務欄位，兩端獨立部署互不阻塞。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 於 research 盤查後產出帳務 G 清單契約文件（欄位、來源、用途、成本），交付 jgb2 端。
2. THE SYSTEM SHALL 以存在性驅動消費 G 欄位：欄位出現即自動啟用分支，缺失走降級，兩端無版本協商。
3. WHERE 現有端點（jgb_bills/bill_detail/payment_logs/invoice_logs/subscription）欄位已足，THE SYSTEM SHALL 優先使用現有欄位，不重複要求擴充。
4. THE SYSTEM SHALL 對個資敏感欄位（繳款人資訊等）定義遮罩輸出規則，且消費端具備非預期格式（如密文）防護。

### Requirement 11：回歸保護

**使用者故事**：作為維護者，我要確保帳務擴充不破壞合約 6 面向、既有單發診斷與檢索行為。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 使合約 6 面向、既有條件診斷 form_fill、售前對話之行為不因本擴充而改變（既有測試全綠）。
2. THE SYSTEM SHALL 擴充進場路由回歸測試（`test_facet_entry_routing_req.py` 模式）：帳務各面向錨點句進對話＋帳務教學/制度問句維持單發，含與合約面向錨點的互不誤吸驗證。
3. THE SYSTEM SHALL 為每個新面向提供可回歸執行的測試：分支決策（決定性單元測試）、對話流程（整合測試）、進場錨點（e2e 口語第一句，走正常管線含 reranker）。
4. WHEN 面向資料與知識部署，THE SYSTEM SHALL 將 reranker semantic model 重建納入部署步驟。
5. THE SYSTEM SHALL 使跨域切換（帳務↔合約）之 e2e 涵蓋雙向各至少一例。
