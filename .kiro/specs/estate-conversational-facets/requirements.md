# 需求規格：estate-conversational-facets（物件管理領域對話面向擴充）

> 建立時間：2026-07-04
> 階段：requirements-generated
> 語言：zh-TW
> 承接：contract／billing／account／iot 四 spec（面向化路徑定型：母子分類＋資料四件套＋領域檔 builder＋face 貫穿＋secondary_calls＋收斂/身分程式保底）。
> 定位：物件是 B 端業者的基本盤功能（建物件→編輯→刊登→招租），為四域外客服案量最大的成塊領域。
> Ground truth：jgb2 物件真碼盤查為 research 階段第一件事（**物件狀態機與各狀態可用功能、編輯/刊登欄位鎖定條件、有合約後的編輯限制、批次上傳格式驗證與失敗原因、對外顯示地址的對內/對外行為**），需求中標注〔待盤查〕者以 research.md 定案為準。
> ⚠️ 素材可靠性警示：官方 help 兩篇狀態機文章（slug39 vs slug5，2021 年）對「洽談中」可用功能描述**互相矛盾**（一寫「新增物件」一寫「查看合約」）——物件狀態機不得沿用文章口徑，一律以現碼定案。

## 簡介

物件領域現況**零成塊知識**（categories 無物件類；建約引導 11 筆屬合約域）、無母子分類、無多輪對話、無系統脈絡疊加。素材證據：客服 C-4 物件 15 案＋C-2 物件 4 案（案型：批次上傳卡住/失敗、編輯資料未儲存、有合約後如何編輯物件、洽談中物件修改限制、格局選項限制、對外顯示地址誤解、招租物件開新頁、進階搜尋欄位、物件刪除後歷史合約去向）＋幫助中心 14 篇已補抓（含 2025-12 新文章「點交點退押金狀態快找」）。年代大宗 2021 舊工單，需以現碼過濾存廢（IoT 先例：預期可轉知識約剩半）。

案型以 **how-to 與狀態機解釋為主、個案診斷為輔**——本領域傾向輕引導（select=category），G 趨近零或僅輕 ground〔jgb2 對外 API 有無 estates/物件端點待盤查〕。

在母分類 `物件管理` 下擴充 2 個子面向〔面向數與名稱由設計定案，變更排障傾向併入面向 1〕：

1. **物件建立與編輯引導**（輕引導）——新增物件、批次上傳（表格填法/失敗原因）、「編輯物件」vs「刊登物件」差異、物件狀態機與各狀態可用功能、有合約後的編輯限制、編輯儲存行為（未按儲存離開會丟失）。
2. **對外曝光與招租店舖**（輕引導）——對外顯示地址（業者後台恆顯完整地址、對外頁才遮蔽的雙層行為）、非刊登中物件廣告頁連結可見性、招租店舖（房東對外首頁）入口/連動編輯/分享。

VR/istaging **不開面向不做知識**（範圍外，識別後導廠商/客服，IoT 先例）；報表匯出、進階搜尋、點交點退快找做**單發知識**（全域功能非對話型）。

擴充原則沿定型路徑：新增面向＝category_config 子分類＋系統脈絡列＋知識掛面向＋對話 config，**引擎零改程式**；若 research 定案需輕 grounding，領域決定性邏輯於 `services/jgb/estates.py`（新領域檔）擴充 builder。

## 名詞定義

- **子面向／輕引導／診斷樹／G 清單**：同前四 spec 定義。
- **物件狀態機**：未刊登/刊登中/洽談中/租約中等狀態與各狀態可用功能的對應〔待盤查；官方文章互相矛盾，以現碼為準〕。
- **編輯 vs 刊登**：「編輯物件」改內容不上架、「刊登物件」直接上架的區分〔欄位鎖定條件待盤查〕。
- **有約後編輯限制**：物件已有合約（待發送/租約中等）時哪些欄位可改、哪些鎖定〔待盤查；客服案「待發送合約要如何修改物件標題」「已建合約不會改物件金額」〕。
- **對外顯示地址**：物件對外廣告頁顯示的遮蔽版地址；業者自己介面仍顯完整地址（客服案高頻誤解點）〔行為細節待盤查〕。
- **招租店舖（房東對外首頁）**：公開展示該業者所有「刊登中」物件的對外頁面，含入口、與會員資訊連動、分享機制。
- **批次上傳**：以表格批次新增物件的功能（表格格式、驗證規則、失敗/卡住原因）〔待盤查〕。

## 範圍

### 範圍內
- 母分類 `物件管理` 下 2 子面向的四件套資料（系統脈絡、對話 config、知識、視盤查結果的 API 設定）。
- 物件狀態機、編輯/刊登欄位鎖定、有約後編輯限制、批次上傳驗證、對外顯示地址行為的 jgb2 真碼盤查與決定性口徑。
- 幫助中心 14 篇素材（已抓）＋客服 19 案的存廢判定與知識產製。
- 單發知識：報表匯出、進階搜尋、點交點退押金狀態快找（2025-12 新功能）。
- 〔待盤查後定〕estates 類對外端點的 client 包裝與輕 grounding（存在性驅動；不存在則純 category 收斂，G 清單記缺口）。
- 進場路由回歸測試擴物件案例（含與合約/IoT/帳號錨點互不誤吸雙向驗證）。

### 範圍外
- 不改對話引擎核心與既有程式保底（全部直接沿用）。
- 不回歸合約 6／帳務 5／帳號 4／IoT 2 面向、售前、既有 form_fill 診斷行為。
- **物件綁電表屬 IoT 域**（已解，scope=switch 共用出口）。
- **物件的合約/點交/押金操作屬合約域**；「點交點退快找」單發知識僅講入口路徑，狀態解讀導合約域。
- **經理人/團隊成員權限屬帳號域**（已解）。
- VR/istaging 範圍外（不對話化、不做知識，識別後導廠商/客服）。
- 資料異動書（改物件名稱等內部作業）不對話化——導客服。
- 不負責 jgb2 端 G 清單（若有）的程式實作與部署時程。

## 需求

### Requirement 1：面向擴充的資料驅動底座

**使用者故事**：作為維護者，我要以純資料方式在 `物件管理` 母分類下掛 2 個新子面向，引擎零改程式即可路由、載入脈絡並進入對話。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 以 `category_config` 子分類（parent=`物件管理`）識別每個新面向：`物件建立與編輯`、`對外曝光與招租`〔面向名由設計定案〕。
2. WHEN 使用者問題命中某面向掛載的知識，THE SYSTEM SHALL 進入該面向的對話 config，行為與前四域進場機制一致。
3. WHEN 面向對話進行中，THE SYSTEM SHALL 注入「通用 base＋母 `物件管理`＋該子面向」三層系統脈絡，不含其他領域層與售前 append。
4. THE SYSTEM SHALL 使新增/調整面向不需修改引擎/route 程式碼（僅資料與 `services/jgb/` 領域檔）。
5. THE SYSTEM SHALL 使每輪注入的系統脈絡總長度不超過既有上限（4500 字）。
6. THE SYSTEM SHALL 為每個對話 config 使用面向專屬 persona_role 鍵（`pm_estate_*` 系列），避免與既有領域鍵互蓋。
7. WHERE 收斂檢索 select=category，THE SYSTEM SHALL 於 grounding_scope 明填 target_user（account 坑防呆先例）。

### Requirement 2：物件建立與編輯引導面向（輕引導）

**使用者故事**：作為管理者，我要建物件、批次上傳、或改物件內容時卡住了，我希望系統依物件目前狀態告訴我能做什麼、怎麼做、為什麼改不了。

#### 驗收標準（EARS）
1. WHEN 使用者詢問新增物件/批次上傳/編輯物件類問題且情境不明，THE SYSTEM SHALL 以至多兩輪分流確認主題（單筆新增？批次？編輯改不了？刊登差異？）。
2. WHEN 情境清楚（問句已含明確主題），THE SYSTEM SHALL 直接收斂作答，不多餘分流（IoT 分流先例）。
3. THE SYSTEM SHALL 依 research 定案的物件狀態機解釋各狀態可用功能；狀態機口徑一律以現碼為準，不沿用互相矛盾的舊文章。
4. WHEN 使用者反映「有合約了物件改不了/怎麼改」，THE SYSTEM SHALL 依 research 定案的欄位鎖定規則說明哪些可改、哪些鎖定與原因〔鎖定條件待盤查〕。
5. ~~批次上傳排查~~（**作廢**——使用者裁定 2026-07-04 批次上傳不處理；persona 識別後直接導客服）。
6. WHEN 使用者反映編輯資料未儲存，THE SYSTEM SHALL 說明編輯儲存行為與正確操作順序〔儲存行為待盤查〕。
7. THE SYSTEM SHALL 不代操作：建立/編輯/刊登/刪除只指路不代執行；刪除類問題須說明影響範圍（如歷史合約去向〔待盤查〕）後導使用者自行操作。

### Requirement 3：對外曝光與招租店舖面向（輕引導）

**使用者故事**：作為管理者，我想知道租客/外部人看到的物件頁長怎樣、對外顯示地址為什麼「沒生效」、招租店舖怎麼用，我希望系統講清楚對內對外的差異。

#### 驗收標準（EARS）
1. WHEN 使用者反映「改了對外顯示地址但還是顯示完整地址」，THE SYSTEM SHALL 解釋對內（業者後台恆完整）/對外（廣告頁遮蔽）雙層行為〔行為細節待盤查〕，並指路確認對外頁的方法。
2. WHEN 使用者詢問非刊登中物件的廣告頁連結可見性，THE SYSTEM SHALL 依 research 定案的可見性規則作答〔待盤查〕。
3. WHEN 使用者詢問招租店舖（房東對外首頁），THE SYSTEM SHALL 說明入口路徑、展示範圍（刊登中物件）、資訊連動（會員中心帳號資訊）與分享方式〔以現行 UI 為準，2021 文章路徑需驗證〕。
4. WHERE 設計定案將本面向併入 Requirement 2 成單一面向，THE SYSTEM SHALL 保持案型覆蓋不變（併面向不減內容）。

### Requirement 4：物件現況輕 grounding（存在性驅動，選配）

**使用者故事**：作為管理者，我問「這個物件現在什麼狀態/能不能刊登」，若系統查得到物件現值，希望直接按這筆物件回答而不是講通則。

#### 驗收標準（EARS）
1. WHEN research 盤查確認 jgb2 對外 API 具物件查詢端點，THE SYSTEM SHALL 定義 client 包裝與收斂 grounding（識別鏈與候選機制沿既有模式）。
2. IF 端點不存在，THEN THE SYSTEM SHALL 以純 category 收斂運作（通則＋指路），並將缺口記入 G 清單（存在性驅動，不阻塞上線）。
3. WHERE 有 grounding，THE SYSTEM SHALL 狀態/日期等一律引用存值，禁止推算；個資遮罩沿既有協定。

### Requirement 5：單發知識與邊界出口

**使用者故事**：作為管理者，我問到報表匯出、進階搜尋、點交點退快找、VR 這類問題時，希望得到正確的單發答案或被導到對的地方，而不是被硬拉進物件對話。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 產製單發知識（不掛面向）：報表匯出、進階搜尋欄位、點交點退押金狀態快找（2025-12 新功能文章為據）。
2. WHEN 使用者問 VR/istaging 類問題，THE SYSTEM SHALL 識別後導廠商/客服，不進入物件面向、不生成 VR 操作知識。
3. WHEN 面向對話中出現跨域完整新問題（綁電表→IoT、合約/點交操作→合約域、經理人權限→帳號域、資料異動→導客服），THE SYSTEM SHALL 判 scope=switch 交由對應領域接手。

### Requirement 6：知識工程與素材治理

**使用者故事**：作為維護者，我要以理解情境的方式產製物件知識，且素材過時/矛盾處以現碼為準修正，經人工閘門後入庫。

#### 驗收標準（EARS）
1. THE SYSTEM（產製流程）SHALL 以幫助中心 14 篇（已抓）＋客服 19 案為素材，逐案存廢判定（轉製/併入/剔除）並留存存廢表。
2. THE SYSTEM SHALL 於素材與現碼衝突時以現碼為準（狀態機矛盾文章、UI 路徑舊稱），修正處記入存廢表證據欄。
3. THE SYSTEM SHALL 使知識 question 用短主題關鍵字、answer 先述情境再帶條件；business_types 依慣例 system_provider；target_user 為 property_manager。
4. THE SYSTEM SHALL 於知識批次入庫前經人工閘門確認（含錨點）。

### Requirement 7：進場路由與誤吸邊界

**使用者故事**：作為維護者，我要物件案例正確進場，且「物件」一詞高頻出現的合約域、綁電表的 IoT 域、經理人的帳號域不被誤吸。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 於進場路由回歸測試新增物件案例（對話型進面向、單發型不進、邊界型進對域）。
2. THE SYSTEM SHALL 雙向驗證誤吸邊界：物件強詞句不吸進合約域（「這份合約的物件」仍屬合約域）、合約/點交操作句不吸進物件域；「綁電表」屬 IoT；「經理人/成員權限」屬帳號域。
3. WHERE SOP 受眾錯位風險存在（已知系統缺口），THE SYSTEM SHALL 以領域強詞錨點規避泛句（IoT 先例），不在本 spec 內解 SOP 隔離。

### Requirement 8：測試與零回歸

**使用者故事**：作為維護者，我要單元/整合/e2e 三層驗證新面向，且四個既有領域零回歸。

#### 驗收標準（EARS）
1. THE SYSTEM SHALL 具備 seed 單元測試（config 形狀、persona_role 專屬鍵、target_user 明填、紅線斷言）。
2. THE SYSTEM SHALL 具備整合測試（真 DB 面向進場、脈絡疊加、候選/收斂流程）。
3. THE SYSTEM SHALL 具備 e2e 測試（真 LLM 多輪：分流→收斂；誤吸邊界負斷言），斷言模式沿「逐輪推進＋全輪累計判定＋機制 token＋紅線負斷言」。
4. THE SYSTEM SHALL 通過既有四域測試全綠（零回歸）。
